## Agnes 视频连续压测

压测方式：低规格文本生成视频，顺序执行 10 次，每次校验任务提交、状态轮询、视频下载和 MP4 文件头。

| 指标 | 结果 |
| --- | ---: |
| 总次数 | `10` |
| 成功 | `9` |
| 失败 | `1` |
| 原始成功率 | `90%` |
| 成功任务平均耗时 | `157.3s` |
| 成功任务最快 | `67.0s` |
| 成功任务最慢 | `266.8s` |

唯一失败：

```text
URLError(SSLEOFError: UNEXPECTED_EOF_WHILE_READING)
```

这属于网关/网络层临时断流。当前产品代码已针对提交、轮询、下载三段加入瞬时错误重试，实际启动器链路会比原始压测脚本更稳。

## 压测暴露的问题

### 1. 视频提交阶段也会变慢

现象：

- 多数提交小于 `1s`
- 个别提交达到 `21.1s`、`31.4s`、`31.5s`
- 授权码视频 smoke 中提交到拿到任务 ID 约 `61.7s`

优化：

- Agnes 提交超时保留 `180s`
- 视频提交阶段增加瞬时错误重试
- UI 在提交阶段显示“正在提交视频任务”，避免用户误判无响应

后续建议：

- 若提交超过 `15s`，UI 可切换为“网关正在接收任务”，减少焦虑感。

### 2. 排队时间波动大

现象：

- 快的任务约 `67s`
- 慢的任务超过 `260s`
- 很多任务长时间只返回 `queued`

优化：

- 后端视频轮询默认上限提升到 `20 分钟`
- 支持 `OPENCLAW_VIDEO_POLL_TIMEOUT_SEC`
- 前端视频 job 持久化，切换模块回来继续显示同一个任务进度

后续建议：

- 增加“任务仍在排队”的小型状态历史，保留最近 3 条状态即可。

### 3. 网关进度字段不稳定

现象：

- 有些任务只有 `queued`
- 有些任务会返回 `in_progress` 和 `progress=30`

优化：

- UI 不依赖百分比作为唯一进度依据
- 后端将状态格式化为产品文案，例如“Agnes 排队中，已等待 104s”“Agnes 生成中，进度 30%，已等待 44s”

后续建议：

- 如果服务商长期只返回 `queued`，前端保持非百分比的呼吸动效，不显示虚假进度条。

### 4. 临时网络断流会造成失败

现象：

- 10 次中 1 次出现 SSL EOF

优化：

- 视频任务提交增加重试
- 视频任务轮询已有连续临时错误重试
- 视频下载增加重试

后续建议：

- 记录失败分类：`submit_transient`、`poll_transient`、`download_transient`、`provider_failed`、`timeout`
- 在诊断包中收敛这些分类，方便售后判断是启动器问题还是网关问题。

### 5. “所有模型”当前不是多模型列表

现象：

- 授权返回了 `defaultModel`、`imageModel`、`videoModel`
- 但 `gatewayModels` 为空

当前结论：

- 三类默认模型均可用
- 没有额外 LLM 模型列表可逐个测试

后续建议：

- 授权后台套餐应明确返回可用模型列表，例如：

```json
{
  "gatewayModels": [
    "agnes-2.0-flash",
    "agnes-2.0-pro"
  ]
}
```

否则启动器只能把 `gatewayDefaultModel` 当作唯一 LLM 模型。
