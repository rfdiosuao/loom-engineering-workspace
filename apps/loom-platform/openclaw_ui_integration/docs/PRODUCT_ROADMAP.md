## 1. 项目定位

OpenClaw U盘启动器不是单纯的“启动脚本套壳”，而是一个面向商家和终端客户的便携式 AI 服务工作台。

核心目标是:

- 用户拿到 U 盘或离线包后，可以直接运行，不依赖本机安装环境。
- 商家可以通过授权码解锁功能、绑定品牌、控制有效期。
- 启动器负责把 OpenClaw、AI 生图、AI 视频、小广告视频制作、API 配置等能力整合成一个易用产品。
- 后续可以扩展成“同一个底座，多套品牌，多种功能包”的商业分发系统。

一句话定位:

**一个可授权、可换皮、可离线分发的 AI 商家工具箱。**

## 2. 当前阶段判断

目前项目已经进入“可交付，但需要稳定化”的阶段。

已具备:

- 桌面启动器界面。
- OpenClaw 本体随包运行。
- Node.js 运行时随包运行。
- Python Bridge 随包运行。
- 包内 Python Runtime，可减少对客户电脑环境的依赖。
- 授权码入口。
- AI 生图、AI 视频、广告视频分镜等业务入口。
- 初步的在线授权服务器和管理后台思路。

- FastAPI native routes 作为 Bridge 主实现，旧 http.server API 路由已退场。
- GitHub Actions 可生成 Release installer/msi。
- 品牌 profile 可在打包时选择 `lumi`、`customer` 或自定义 profile。

主要短板:

- 架构经历过多次抢救式修改，需要整理成稳定版本。
- 授权服务器还需要正式产品化。
- GitHub installer/msi 已自动化，但客户 portable zip 仍需交付前验收。
- 真实 U盘授权拔插、不同 Windows 机器安装启动还需要硬件和环境验证。
- 授权后台、机器人绑定、Skills 产品化仍待推进。
- UI 虽然可用，但还需要继续沉淀统一设计规范和客户交付说明。

## 3. 发展总路线

项目后续建议分四个阶段推进。

### 阶段一: 稳定交付版

目标:

把当前版本稳定成可以销售和交付的“基础商业版”。

重点工作:

- 固定离线包目录结构。
- 固定启动流程: 启动器 -> Bridge -> OpenClaw -> 本地网页。
- 固定端口策略和进程清理策略。
- 保证无 Python、无 Node.js 的客户电脑也可以运行。
- 授权页、API 配置页、服务日志页必须稳定。
- 每次发包前执行统一验收清单。

阶段完成标准:

- 新电脑解压后双击即可启动。
- 未授权状态提示清楚。
- 输入有效授权码后可以解锁。
- 配置 API Key 后重启仍然生效。
- OpenClaw 页面可以正常打开。
- 日志不循环刷屏。
- zip 包内不包含个人授权、个人 API Key、测试缓存。

建议周期:

1 到 2 周。

---

### 阶段二: 商业授权版

目标:

让项目具备正式售卖能力，而不是手工给客户发包。

重点工作:

- 完善授权码服务器 Web UI。
- 支持一份安装包对应一个授权码。
- 支持授权码状态: 未使用、已激活、已禁用、已过期。
- 支持授权码绑定设备或安装 ID。
- 支持商户管理: 名称、Logo、主题色、联系方式、功能权限。
- 支持在线激活和离线校验。
- 后台支持查询客户激活记录。

阶段完成标准:

- 后台可以生成授权码。
- 授权码可以导出给销售或客户。
- 客户输入授权码后，启动器自动保存授权状态。
- 同一个授权码是否允许换设备，有明确规则。
- 授权到期、禁用、无网络等情况有清晰提示。

建议周期:

2 到 4 周。

## 6. 授权码策略建议

授权码建议继续采用“在线激活 + 本地签名凭证”的模式。

推荐规则:

- 一份安装包可以对应一个授权码，也可以同一个安装包配多个授权码销售。
- 授权码由服务器随机生成。
- 激活后服务器返回签名授权文件。
- 客户端本地只保存签名结果，不保存服务器私钥。
- 客户端启动时优先本地验签，必要时再联网刷新状态。

授权码不要只写死在启动器里。

原因:

- 写死在启动器里不方便销售和撤销。
- 每次换客户都要重新打包。
- 后期无法统计激活和到期。

正确做法:

- 启动器只负责输入和校验授权码。
- 授权码生成、状态管理、商户绑定都放到服务器后台。

## 7. UI 和主题发展方向

UI 模块必须设计成可替换。

推荐方向:

- 基础组件统一: 按钮、输入框、侧边栏、卡片、弹窗、提示。
- 业务页面和视觉主题分离。
- 主题配置支持:
  - 产品名称。
  - Logo。
  - 主色。
  - 辅助色。
  - 背景色。
  - 字体色。
  - 左侧菜单项。
- 商户激活后自动切换对应主题。

长期目标:

**不要为了每个客户改代码，而是让授权码决定这个客户看到什么品牌和功能。**

## 8. 广告视频模块规划

广告视频模块是未来最有商业价值的功能之一。

建议把它做成“广告视频工作台”，而不是普通的视频生成按钮。

核心能力:

- 分镜列表: 每个镜头独立文案、画面、动作、时长。
- 三视图: 正面、侧面、俯视或产品三角度参考。
- 首尾帧: 控制视频开始和结束画面。
- 九宫格: 对比不同提示词、构图、风格。
- 提示词拆分:
  - 产品描述。
  - 场景描述。
  - 镜头运动。
  - 光线风格。
  - 字幕文案。
- 模板化:
  - 餐饮广告。
  - 本地生活。
  - 美业。
  - 课程引流。
  - 产品展示。

阶段目标:

- 第一版: 可编辑分镜 + 首尾帧 + 九宫格。
- 第二版: 模板库 + 历史记录。
- 第三版: 自动生成完整广告方案。

## 9. 打包和交付规范

每次交付前必须检查:

- 是否包含 `OpenClaw.exe`。
- 是否包含 `node/node.exe`。
- 是否包含 `node_modules/openclaw`。
- 是否包含 `_up_/python/bridge.py`。
- 是否包含 `_up_/python-runtime/python.exe`。
- 是否不包含个人 `license.json`。
- 是否不包含个人 `install_id.txt`。
- 是否不包含个人 API Key。
- 是否清理 `__pycache__` 和 `.pyc`。
- 是否可以在非开发目录启动。
- 是否能在无系统 Python 的环境启动。

建议建立两个包:

- `OpenClaw-Portable-dev`: 内部测试包，可以保留测试配置。
- `OpenClaw-Portable-release`: 对外交付包，必须干净。

## 10. 测试验收清单

每次发版至少测试这些场景:

- 解压后直接双击启动。
- 未授权状态显示正常。
- 输入错误授权码，提示正常。
- 输入正确授权码，激活成功。
- 重启后授权仍然存在。
- 启动服务后 OpenClaw 页面能打开。
- OpenClaw 不要求 token。
- API 配置保存后，重启仍然生效。
- 生图功能能读取 API 配置。
- 视频功能失败时能给出清晰提示。
- 服务日志不会无限循环刷屏。
- 关闭启动器后相关进程能释放。
- 换目录运行不依赖固定路径。
- zip 包内无个人敏感数据。

## 11. 协作和 Agent 编排建议

后续如果继续使用多个模型或多个 Agent，必须拆成小任务，不能让一个模型“全项目随便改”。

推荐角色:

- 架构负责人: 只定边界和验收标准。
- 启动链路 Agent: 只负责 Tauri、Bridge、OpenClaw 进程。
- UI Agent: 只负责 React 页面和样式。
- 授权服务器 Agent: 只负责授权码后台和接口。
- 打包 Agent: 只负责离线包、清理、压缩、验收。
- 测试 Agent: 只跑测试清单，不随便改代码。

每个 Agent 必须遵守:

- 只改自己负责的目录。
- 修改前先读现有文档。
- 修改后必须说明改了哪些文件。
- 必须跑对应验证命令。
- 不允许删除别人的改动。

## 13. 最重要的原则

这个项目接下来不要再走“哪里坏了修哪里”的路线。

必须坚持三条原则:

- **先稳定交付，再做大重构。**
- **先把商业闭环跑通，再追求架构漂亮。**
- **所有可变内容都配置化，不为单个客户硬改代码。**

只要按这个节奏推进，项目可以从现在的抢救状态，逐步变成真正可销售、可维护、可扩展的产品。
