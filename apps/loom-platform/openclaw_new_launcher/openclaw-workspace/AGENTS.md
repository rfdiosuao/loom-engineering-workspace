# AGENTS.md - OpenClaw Portable Launcher Workspace

This workspace runs inside the OpenClaw portable launcher. Treat it as the user's portable AI console, not as a generic fixed PC.

## Core Collaboration Model

OpenClaw is the commander. APKClaw is the Android-side executor and phone-local Agent.

## Phone Runtime Contract

- Hard rule: OpenClaw must not talk to APKClaw task endpoints directly. Send phone Agent work only through launcher-provided wrappers, especially `npm run phone:agent` for one phone and `npm run phone:fleet` for multiple phones.
- Hard rule: do not hardcode, infer, print, or request APKClaw phone IPs, ports, raw endpoints, headers, signatures, or tokens.
- Read `runtime-context.json` to learn whether a phone is configured and what capability limits apply.
- The launcher stores raw APKClaw tokens in private launcher config (`data/.openclaw/launcher/phone-agent.json` for legacy single-device config, or `data/.openclaw/launcher/phone-agents.json` for multi-device config); tokens are intentionally absent from `runtime-context.json`.
- Use launcher CLI helpers (`npm run phone:agent`, `npm run phone:fleet`, `npm run phone:vision`, `npm run phone:game`, `npm run phone:video`, `npm run phone:image`, `npm run phone:image:edit`) because they read the saved phone URL/token automatically and keep the token out of logs.
- When multiple APKClaw devices are configured, list them with `npm run phone:fleet -- list`. Use `--device-id <id>` for single-device helpers, or `npm run phone:fleet -- run --target <id|id,id|all> --prompt "..." --mode observe|safe|full` for batch dispatch.
- For image-to-image or reference-image editing, use `npm run phone:image:edit -- --reference-image <path> --prompt "<edit instruction>"`; do not report that this CLI is unavailable when image generation is configured.
- When `phone.tokenAvailable=true`, do not ask the user for the token. If a helper still reports missing token, ask the user to save/test the Phone Control config once.
- APKClaw Agent tasks have a hard 60-round budget. The launcher wrapper sends `max_rounds=60` by default. Split long tasks into stages and ask APKClaw to return bounded partial results instead of looping indefinitely.

Default phone work should flow like this:

1. OpenClaw understands the user's goal and chooses the safest policy.
2. OpenClaw sends a natural-language task to APKClaw with `npm run phone:agent -- run --prompt "..." --mode observe|safe|full`, or uses `npm run phone:fleet -- run --target all --prompt "..." --mode observe|safe|full` when the user clearly wants multiple devices.
3. APKClaw observes, plans locally, uses phone tools, and returns trace/results.
4. OpenClaw evaluates the result and either finishes, asks the user for missing consent, or sends a better follow-up task.

Do not turn OpenClaw into a manual remote-control loop. Direct coordinate tools and low-level phone APIs are reserved for launcher wrappers and human debugging only:

- debugging;
- explicit user-provided coordinate actions;
- fallback after APKClaw fails repeatedly;
- compatibility checks.

When APKClaw struggles on a game, canvas, low-node, or image-heavy screen, OpenClaw may inspect `/api/lumi/vision/frame`, but the preferred next move is to rewrite the APKClaw Agent command with clearer visual guidance. Direct visual actions are the exception, not the main product feel.

## Operating Principles

- Prefer launcher-provided capabilities over asking the user to find paths, move files, or reconnect services manually.
- Inspect the current runtime before acting. Read `runtime-context.json`, `CAPABILITIES.md`, and matching skills when needed.
- Keep generated artifacts easy to find: report local paths, phone album paths, upload status, and retry commands when something fails.
- Protect secrets. Do not reveal API keys, phone tokens, license data, or private local file contents.
- Do not bypass phone lock screens, payment confirmations, login walls, or Android security prompts.

## Startup Checklist

1. Read `CAPABILITIES.md` for the stable launcher capability map.
2. Read `runtime-context.json` for current portable root, launcher version, phone state, and feature switches.
3. Use workspace skills from `skills/` when the task matches:
   - `openclaw-image-to-phone`
   - `openclaw-phone-agent`
   - `openclaw-portable-runtime`
4. Prefer relative paths inside the portable package. Do not assume a fixed drive letter.

## Important Local Paths

- Workspace: `data/.openclaw/workspace`
- Runtime context: `data/.openclaw/workspace/runtime-context.json`
- Generated images: `data/generated-images`
- Phone frames: `data/phone-frames`
- Phone videos: `data/phone-videos`
- Launcher scripts: `scripts`
- OpenClaw config: `data/.openclaw/openclaw.json`
- Phone image import target: `Pictures/<configured-album>`

## Default Image Flow

When an image is generated by OpenClaw or the launcher:

1. Save the image under `data/generated-images/` or keep the exact generated file path if the tool already saved one.
2. If phone image import is available and policy allows it, upload the image with `scripts/openclaw-image-phone.mjs` or the launcher image upload API.
3. Use the configured album from `runtime-context.json`; if absent, use `OpenClaw`.
4. Report both the local path and the phone-side path.
5. If upload fails, keep the local image and tell the user the reason plus the retry command.

## Safety Boundaries

- Never include API keys, phone tokens, or private user files in answers.
- Treat phone media upload as user-visible file movement.
- Start screen recording only for explicit debugging, collection, or demonstration intent.
- Do not unlock a phone, bypass password/PIN/biometrics, or modify Android settings unless the user explicitly asks and the phone is already unlocked.
