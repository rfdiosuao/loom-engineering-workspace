# 麓鸣获客工作台真实落地说明

日期：2026-07-07
分支：codex/customer-acquisition-v1

## 这套系统到底要做什么

麓鸣获客工作台的目标不是“做一个看起来很厉害的获客 UI”，也不是直接伪装成平台官方接口去发私信、评论或加好友。

真正的产品形态是：

```text
获客工作台 UI
-> 本地 Bridge / CLI
-> 麓鸣手机 Agent / 多设备调度
-> 真实手机 App 内读取公开线索、填入草稿、停在确认页
-> 手机 Agent 返回结构化 JSON
-> 后端沉淀线索池、客户池、草稿、跟进日志
-> 飞书 CLI 写入飞书多维表格
```

用户看到的是一个“AI 私域获客助手 / 手机端智能销售助理 / 本地生活商家线索助手”。底层可以使用手机控制、Matrix、多设备调度、发布、桌面回复、日志、飞书 CLI，但对外不能包装成脚本、群控、外挂或自动骚扰工具。

## 不要走偏的产品边界

允许默认实现：

- 读取用户已登录、已授权手机 App 里可见的公开内容。
- 根据行业、地域、关键词生成获客任务。
- 把评论、搜索结果、主页信息、私信会话摘要等整理为线索。
- AI/规则判断线索质量，写入线索池和客户池。
- 生成评论回复、私信、微信跟进、二次跟进草稿。
- 将草稿填入手机 App 或桌面输入框，但默认停在人工确认页。
- 人工确认后记录“已手动发送”“客户回复”“下一步跟进”。
- 写入飞书多维表格，飞书不可用时进入 pending sync。

禁止默认实现：

- 无确认批量私信。
- 无确认批量评论。
- 无确认批量加好友或加微信。
- 自动骚扰式群发。
- 绕过验证码、平台风控或封禁策略。
- 保存真实账号密码、token、Gitee 令牌、生产客户隐私数据。
- 用真实客户数据做演示或测试。

所有真实对外动作都必须默认经过：生成草稿、人工确认、白名单、频控、日志留痕。

## 当前已具备的真实能力

根据当前分支代码，已有这些基础能力可以继续复用：

- UI 已有“获客”入口和工作台页面。
- 后端已有获客 snapshot、demo flow、真实线索导入、草稿确认、人工发送记录。
- 线索导入支持 CSV、评论 JSON、剪贴板文本等形式。
- 后端会做基础去重、规则评分、客户池沉淀、草稿生成、任务日志。
- 飞书集成已有 doctor、status、login、bind-table、create-table、test-write、retry-sync 等接口。
- 模板沉淀已有本地保存、上传、重试入口。
- 手机控制侧已有 phone Agent、Bridge、Matrix、发布、日志等基础模块可接。

这些说明：获客工作台不是从零开始。下一步应该把“手机 Agent 结果入库”和“飞书真实写入”做扎实，而不是继续堆静态营销文案。

## 当前主要债务

### P0：手机 Agent 获客任务链路还没彻底闭环

目标形态：

```text
UI 点击创建手机 Agent 获客任务
-> POST /api/matrix/acquisition/agent/run
-> loom_cli.py acquisition agent-run ...
-> 手机 Agent dry-run 或真机执行
-> 返回 loom.acquisition.agent_result.v1
-> 后端 ingest
-> 线索池 / 客户池 / 草稿队列 / 日志 / 飞书同步更新
```

验收标准：

- dry-run 能返回结构化结果并进入线索池。
- 真机路径必须停在人工确认页。
- 结果包含 taskId、deviceId、platform、action、status、leads、drafts、logs。
- 日志中能看到 agent.result_ingested。

### P0：飞书多维表格必须成为真实客户池

目标形态：

```text
线索入库
-> 字段映射
-> 飞书 CLI 写入多维表格
-> 写入成功返回 synced
-> 失败进入 pending_sync
-> 用户可重试同步
```

验收标准：

- 新表创建可用。
- 已登录飞书后真实写入一条线索。
- 写入字段包含来源平台、客户昵称、线索内容、意向评分、跟进草稿、状态、下一次跟进时间、任务 ID。
- 不把 token、cookie、账号密码写入状态文件或日志。

### P1：UI 里仍有一部分“能力展示”需要变成真实按钮

需要逐项把营销感文案落到真实动作：

- “自动找线索”必须对应手机 Agent 任务或真实导入入口。
- “AI 写话术”必须对应草稿生成接口和知识库/SOP。
- “沉淀客户池”必须对应本地状态和飞书写入。
- “多台手机矩阵”必须对应设备选择、任务队列、失败隔离、频控和日志。
- “自动上传模板”必须对应模板库上传接口、失败重试、服务器后台可查看。

### P1：模板云端沉淀需要后台和权限模型

目标形态：

- 每个安装用户默认上传自己沉淀的模板。
- 模板上传前做脱敏，不上传账号、客户隐私、token、cookie。
- 服务器后台能看到模板列表、来源版本、行业、适用平台、上传时间、状态。
- 上传失败进入本地 pending queue。
- 后台只展示模板，不展示用户私密线索。

### P2：海外平台发布与获客需要明确真机策略

TikTok、RedNote、Lemon8 等海外平台优先通过真实手机 App 路径做：

- 手机上登录平台账号。
- 手机网络和代理由用户自行配置。
- 电脑只通过局域网、USB ADB forward 或可达的手机 Agent 通道发任务。
- 如果手机开代理导致局域网 IPv4 不通，优先使用 USB ADB forward、同网段旁路、代理分流、IPv6 可达性验证，而不是强行绕平台风控。

## 推荐的最小可演示闭环

第一阶段只做一条稳定路径：

```text
1. 工作台创建获客任务
2. 选择平台、行业、设备、SOP、飞书表
3. 点击“手机 Agent dry-run”
4. 后端生成或接收 agent_result JSON
5. 入库为真实线索
6. 生成跟进草稿
7. 草稿进入人工确认状态
8. 写入飞书多维表格
9. UI 显示线索、客户、草稿、日志、飞书状态
```

第二阶段再接真机：

```text
手机 Agent 打开目标 App
-> 搜索/读取公开内容
-> 识别潜在线索
-> 填入草稿
-> 停在确认页
-> 返回结构化结果和截图/日志
```

第三阶段再做多设备：

```text
设备池
-> 任务队列
-> 每台手机频控
-> 失败隔离
-> 结果聚合
-> 飞书统一客户池
```

## 给 Codex 的实现判断

当后续 Codex 继续开发时，按这个判断：

- 如果只是新增漂亮卡片，但没有接口、状态、测试、飞书写入，不算真实落地。
- 如果只是本地模拟，但不能沉淀到线索池、草稿、日志，不算闭环。
- 如果能 dry-run 入库、能写飞书、能在 UI 看到状态，算第一阶段可用。
- 如果能通过手机 Agent 在真实 App 填草稿并停在确认页，算真机闭环。
- 如果能多设备调度、频控、失败隔离、统一写入飞书，算矩阵放大。
