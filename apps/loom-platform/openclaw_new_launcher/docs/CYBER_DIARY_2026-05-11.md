# 赛博日记：2026-05-11

## 今天最值得留下的判断

我们的初衷不是把 APKClaw 做成一个越来越重的“大脑”，而是采用 Agent 协作指挥 APKClaw 的方案。

更清晰的架构是：

- OpenClaw：计划、判断、分工、复盘。
- 启动器：控制台、运行时、配置、素材中转。
- 专门 Agent：视觉、采集、生成、测试、文档、演示。
- APKClaw：手机端执行器，负责观察、点击、输入、滑动、录屏、回传状态。

这条线很重要。只要守住它，后面加 OCR、VLM、采集器、游戏辅助、素材流转，都不会把手机 APK 做成一团难维护的东西。

## 今天形成的方法论

1. 先分清“脑”和“手”

真正的产品体验来自协作：Agent 先想清楚任务，再调用 APKClaw 执行。APKClaw 不需要理解所有商业场景，它需要把手机能力暴露得稳定、可诊断、可回放。

2. 采集不是只要拿到节点

微信、QQ、游戏、WebView、商城评论区这类场景提醒我们：无障碍树只是一个通道，不是全部世界。没有节点时要明确返回 `no_accessibility_nodes` 和 `needsVision=true`，让上层 Agent 切到截图、OCR、VLM。

3. 错误码就是产品体验

“没抓到”没有价值。“前台包不对”“采集器忙”“敏感应用已脱敏”“需要视觉模式”才有价值。好的错误会让 Agent 知道下一步怎么恢复。

4. 素材闭环会让产品更像生产工具

PC 生成图片传到手机，是从电脑到手机；手机录屏回传电脑，是从手机到电脑。两边打通后，AI 可以生成素材，也可以收集演示、失败过程、用户路径，再做剪辑、标注和复盘。

5. 隐私不是以后补的东西

QQ、微信、相册、文件、支付、银行这类内容必须默认保护。APKClaw 可以观察结构和状态，但不应该随便吐出私人文本。商业化越往前走，这条越重要。
