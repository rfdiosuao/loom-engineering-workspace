# 麓鸣中枢智能体原生输入区、模型切换与能力自动编排设计

日期：2026-07-17

产品基线：LOOM 2.1.89 Hotfix

状态：产品设计与完整能力继承范围已确认，待实施

决策人：麓鸣产品负责人

## 1. 文档定位

本文档是以下既有设计的增量修订：

- `2026-07-15-loom-agent-matrix-production-design.md`
- `2026-07-16-loom-native-vertical-agent-design.md`

本文明确覆盖旧设计中以下内容：

- `AgentComposer` 不再常驻展示“设备”“设备组”和“能力”三个技术控件。
- 普通用户不再手动勾选 Skill、MCP 或 CLI 能力。
- `capabilityHints` 不再作为正式产品交互；旧客户端字段仅保留兼容读取。
- 模型切换从全局设置页延伸为会话级选择，并保留显式“设为默认模型”命令。
- 未手动选择设备范围不再直接等同于矩阵任务失败；服务端先做自动范围解析，再决定执行或在对话中澄清。
- 顶部不再提供重复的“新对话”按钮，也不再常驻暴露开发者调试按钮。
- 用户发送消息后，在首个 Agent 可见事件到达前显示“正在思考”状态。

未被本文覆盖的会话、运行、策略、审批、矩阵租约、Recipe、证据和恢复设计继续有效。

## 2. 目标

把中枢智能体从“暴露技术参数的工具面板”改成真正的麓鸣原生 Agent：

1. 用户只描述目标，智能体自行选择已接入的全部麓鸣能力。
2. 用户可以像 Codex 一样在输入区切换当前对话模型。
3. 多手机目标默认由自然语言和矩阵状态自动解析。
4. 普通界面只显示业务动作和结果，内部能力 ID、权限代码和原始错误进入调试器。
5. 保留现有权限、TaskGrant、矩阵租约和硬停止规则，不用“简化 UI”换取越权执行。

## 3. 已确认产品决策

| 决策项 | 结论 |
|---|---|
| 模型作用域 | 默认仅对当前对话生效 |
| 全局默认 | 模型菜单提供独立“设为麓鸣默认模型”命令 |
| 能力选择 | 从普通输入区删除，由 Agent 自动选择 |
| 能力覆盖 | 所有可用 Internal、MCP、CLI、Skill 均进入原生 Agent 能力体系 |
| 核心能力闭环 | 生图、图片编辑、视频生成、单机读取与控制、多机矩阵控制必须由原生 Agent 实际调用成功 |
| 设备范围 | 默认自动解析；需要时通过紧凑“执行范围”入口手动覆盖 |
| 视觉方向 | 保留麓鸣浅色品牌，借鉴 Codex 的层级和输入区交互，不复制其深色皮肤 |
| 技术信息 | 普通视图中文化；原始能力 ID 和协议错误仅在调试器显示 |
| 新对话入口 | 仅保留会话栏搜索框右侧的 `+`，删除页头重复按钮 |
| 调试入口 | 从页头移除，下沉到单次运行的“运行详情”和开发者入口 |
| 等待反馈 | 发送后立即显示“正在思考”，首个可见运行事件到达后切换为真实内容 |

## 4. 交互方案

### 4.1 页面结构

中枢智能体保持三段式结构：

```text
顶部：身份、连接状态、当前模型摘要
中部：会话列表 + 对话主舞台
底部：原生 Agent 输入区
```

对话仍是唯一主舞台。模型、范围和附件属于当前消息上下文，不新增常驻设置侧栏，不把输入区做成表单。

页头遵循单一入口原则：

- 删除右上角 `</>` 调试按钮。
- 删除右上角“新对话”按钮。
- 新建对话统一使用会话栏搜索框右侧的 `+`，避免同一命令在同一视图重复出现。
- 页头右侧不留下空白按钮容器；身份和状态信息自然占据可用宽度。
- 运行 ID、能力数量和授权数量不作为页头主信息，只有确有诊断价值时才进入运行详情。

### 4.2 新输入区

输入区结构：

```text
┌──────────────────────────────────────────────────────────────┐
│ 输入任务或问题                                               │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│ ＋附件   ◎ 自动范围                    当前模型 ▾   发送图标 │
└──────────────────────────────────────────────────────────────┘
```

规则：

- 删除常驻“设备”文本框。
- 删除常驻“设备组”文本框。
- 删除“能力”下拉和英文能力列表。
- 附件按钮使用图标，保持至少 40×40 像素点击区域。
- “执行范围”显示当前解析状态：`自动`、`全部在线`、`招聘一组`、`P01、P02` 或 `2 台设备`。
- 模型选择位于发送按钮左侧，显示当前模型短名称。
- 发送使用熟悉的向上箭头图标；运行中切换为停止按钮。
- `Enter` 发送，`Shift+Enter` 换行；保留输入法组合状态判断。
- 输入框随内容增长但设置稳定最小和最大高度，不挤压历史消息。

### 4.3 模型菜单

模型菜单参考 Codex 的紧凑层级，但只展示麓鸣账号当前可用的文本模型：

```text
模型
  ● qwen3.7-plus            当前对话
    glm-4.5
    deepseek-v3.2
  ─────────────────────
  设为麓鸣默认模型
  管理模型
```

行为：

- 选择模型后立即更新当前会话，不修改其他会话。
- 新会话继承麓鸣全局默认文本模型。
- “设为麓鸣默认模型”显式调用现有账号模型选择接口。
- “管理模型”跳转现有模型页面。
- 运行已经开始后，切换只影响下一条消息，不修改正在运行的请求。
- 不展示未验证为文本模型的图像、视频或手机观察模型。
- 模型不可用、账号离线或同步过期时保留当前选择并给出中文修复动作。

### 4.4 “正在思考”反馈

用户点击发送或按下 `Enter` 后，界面必须在 100 毫秒内给出反馈。用户消息落入对话流后，在其下方显示一条紧凑的 LOOM 状态行：

```text
LOOM  ● ● ●  正在思考
```

状态规则：

- 本地发送状态建立后立即显示，不等待服务端返回 `runId`。
- 收到 `run.queued` 或 `run.started` 时继续显示，并转由服务端运行状态托管。
- 收到首个普通用户可见事件时移除占位并切换为真实内容。可见事件包括 `message.delta`、`message.completed`、`plan.updated`、`tool.started`、`approval.required` 和可展示的矩阵进度。
- 如果首个终态是 `run.failed` 或 `run.cancelled`，先移除思考状态，再在同一位置显示中文错误或取消结果，不留下无限动画。
- 切换会话、归档会话或用户停止运行时，不得把上一会话的思考状态带入当前会话。
- 同一会话同一时刻只显示一个思考状态，重连和重复事件不能生成多个占位。

动效规则：

- 三个圆点依次以 `opacity` 和轻微 `transform` 呼吸，单轮 900-1200 毫秒，不动画宽高，不引发布局跳动。
- 状态行预留稳定高度，真实内容出现时使用 150-200 毫秒淡出，不移动已有消息。
- `prefers-reduced-motion: reduce` 下关闭位移和循环动画，仅保留静态圆点与“正在思考”文字。
- 状态容器使用 `role="status"` 和 `aria-live="polite"`，读屏只播报一次“麓鸣正在思考”，不逐个播报圆点。

### 4.5 执行范围

“执行范围”不是普通聊天的必填项。默认值为 `auto`。

点击范围入口后显示真实矩阵数据，不使用自由文本猜测设备：

- 自动识别。
- 所有在线设备。
- 已存在设备组。
- 可搜索并多选具体设备。

自动解析规则：

- “所有手机”“全部在线设备”解析为全部在线设备。
- 明确设备 ID 如 `P01、P02` 解析为对应真实设备。
- 明确设备组如“招聘一组”解析为对应真实组。
- 单设备上下文明确时可继承当前聚焦设备。
- 普通问答、状态解释和不涉及手机的任务不绑定设备。
- 目标存在多个可能匹配或用户未表达批量范围时，在对话中提出一次简短澄清，不弹技术错误。

自动范围解析必须在执行矩阵写操作前编译为确定的 `TaskGrant` 和 `requestScope`。模型不能绕过该步骤直接扩大范围。

## 5. 会话级模型架构

### 5.1 数据模型

`AgentSession` 增加可选字段：

```ts
type AgentSession = {
  // existing fields
  modelId?: string;
};
```

语义：

- 有 `modelId`：当前会话固定使用该模型。
- 无 `modelId`：继承账号当前默认文本模型。
- 每次创建运行时把最终模型写入运行快照，保证历史可追踪。

`AgentRun` 增加：

```ts
type AgentRun = {
  // existing fields
  modelId?: string;
  modelSource?: 'session' | 'account-default';
};
```

### 5.2 API 演进

`GET /api/agent/bootstrap` 增加：

```json
{
  "models": [
    {"modelId": "qwen3.7-plus", "name": "qwen3.7-plus", "available": true}
  ],
  "defaultModelId": "qwen3.7-plus"
}
```

`POST /api/agent/sessions` 和 `PATCH /api/agent/sessions/{sessionId}` 接受可选 `modelId`。

`POST /api/agent/sessions/{sessionId}/messages` 不依赖前端重复传模型；服务端从会话读取并固化到运行请求。为兼容乐观新会话，允许请求携带同值 `modelId`，但服务端必须验证并以会话记录为准。

### 5.3 服务端验证

- 模型必须属于当前账号公开的可用文本模型集合。
- 模型 ID 只保存名称，不保存 Token、Provider 密钥或完整网关配置。
- `LoomModelClient` 从当前受保护账号会话读取网关地址和令牌，再用运行快照中的 `modelId` 覆盖本轮 `profile.model`。
- 模型切换不得调用全局 `select_models`；只有“设为默认”命令可以修改账号默认。
- 模型在会话创建后被账号移除时，发送前返回稳定的 `AGENT_MODEL_NOT_AVAILABLE`，并提供回退到默认模型的操作。

## 6. 能力自动编排

### 6.1 产品规则

用户不选择能力。Agent 每轮都获得当前可用能力的结构化定义，并根据目标自动选择。

能力来源完整保留：

- `internal`：矩阵、媒体、日志等直接内部服务。
- `mcp`：麓鸣内置 MCP 及已受信 MCP。
- `cli`：允许列表中的确定性 LOOM CLI 命令。
- `skill`：官方与用户启用的声明式 Skill。

“完整继承”不是仅在 Bootstrap 中出现能力名称。每项可用能力必须同时满足：

1. 能力目录能够发现该能力并给出稳定 canonical ID。
2. 能力具备已连接的真实执行器，不允许只注册空壳定义。
3. 输入 Schema、中文名称、中文描述、来源、权限和风险元数据完整。
4. 每轮模型请求自动注入该能力，不依赖用户勾选，也不受旧 `capabilityHints` 限制。
5. 模型返回的网关工具别名能够还原为 canonical ID，并由同一注册表执行。
6. 执行继续经过 Policy、TaskGrant、矩阵租约、审批、超时、取消和急停边界。
7. 结果以对话消息、媒体结果或矩阵任务附件回显，失败提供中文恢复动作。

首批必须完成真实闭环的内置能力族：

| 能力族 | 必须继承的能力 |
|---|---|
| 图片 | 文生图、多图生成、图片编辑、尺寸/比例和模型参数、生成结果入媒体库 |
| 视频 | 文生视频、图生视频、模型与时长等受支持参数、生成任务和结果入媒体库 |
| 单机手机 | 状态、截图、屏幕读取、受控任务执行、停止与结果证据 |
| 多机矩阵 | 状态、范围解析、分发、进度、屏幕、暂停/继续、取消、失败重试和急停 |
| 麓鸣系统 | 状态、模型、任务、日志和诊断等已允许的确定性能力 |
| 扩展能力 | 已安装且启用的 Skill、麓鸣内置及受信 MCP、允许列表内 CLI |

以后新增用户可调用的麓鸣功能时，功能注册必须同时声明 Agent 暴露元数据；合同测试比较“可调用产品能力清单”和 Agent 已连接目录，任何未声明的遗漏都使测试失败，避免长期依靠手工白名单同步。

完整继承不等于无条件授权。不可用能力应带明确 `available: false` 和中文原因；高风险能力仍可进入目录，但执行时必须通过现有审批和安全边界，模型和系统提示词都不能绕过。

`capabilityHints` 处理方式：

- 前端停止发送。
- 后端继续接受一个发布周期，防止旧客户端失败。
- 编排器不把它当成白名单，也不能据此隐藏其他能力。
- 后续协议版本删除该字段。

### 6.2 中文能力元数据

能力摘要增加：

```json
{
  "name": "loom.mcp.loom.loom_status",
  "displayName": "读取麓鸣状态",
  "description": "读取当前麓鸣、模型账号、组件和控制面状态",
  "domain": "system",
  "source": "mcp",
  "permission": "read"
}
```

要求：

- 内置能力必须提供中文 `displayName` 和 `description`。
- 未本地化的第三方能力显示可读名称和来源，不在普通界面展示全大写权限代码。
- 原始 `name` 只用于协议、审计和调试。
- 工具名发送到 OpenAI 兼容网关前编码为合法、唯一、最长 64 字符的别名；响应后按本次请求映射回 canonical 能力 ID。

### 6.3 系统提示词

原生 Agent 系统提示词改成中文能力编排合同，至少包含：

1. 你是麓鸣原生中枢智能体，不是外部 Agent。
2. 根据用户目标自主选择本轮已提供能力，不要求用户勾选工具。
3. 只调用请求中真实存在的工具，禁止编造能力名。
4. 普通状态查询优先使用只读内部或 MCP 能力。
5. 多机任务先读取矩阵状态、解析范围，再分发任务。
6. 确定性内部能力优先；MCP、CLI 和 Skill 作为同一注册表中的受控工具使用。
7. 工具失败后根据错误调整方案，不得原样无限重复同一失败调用。
8. 对用户使用简体中文解释目标、进度、阻塞和结果，不暴露内部能力 ID。
9. 外发、高风险和硬停止动作继续服从 Policy 与 TaskGrant，提示词无权放宽。

模型工具定义仍使用结构化 Schema；不能只把能力列表拼成自由文本后让模型输出命令。

## 7. 自动范围与安全边界

新增 `scopeMode`：

```json
{
  "scopeMode": "auto",
  "targets": {"deviceIds": [], "groups": []}
}
```

取值：

- `auto`：服务端根据用户文本、会话上下文和矩阵事实解析。
- `manual`：使用用户在范围菜单中选定的稳定设备 ID 和组 ID。

自动解析输出：

```json
{
  "status": "resolved",
  "source": "prompt",
  "deviceIds": ["P01", "P02"],
  "groups": ["招聘一组"],
  "allOnline": false,
  "evidence": ["用户明确提到招聘一组"]
}
```

状态可以是 `not_required`、`resolved` 或 `ambiguous`。

- `not_required`：继续普通 Agent 对话。
- `resolved`：生成 TaskGrant 后允许矩阵工具使用该范围。
- `ambiguous`：Agent 返回澄清问题，不创建矩阵写任务。

模型提出的工具参数只能缩小已解析范围，不能扩大范围。`所有在线设备` 必须来自用户明确表达或手动选择，不能由模型自行推断。

## 8. 普通视图与调试视图

### 8.1 普通视图

- 工具调用显示“正在读取麓鸣状态”“正在检查设备”“正在分发任务”等中文动作。
- 成功工具块默认折叠为一行，可展开查看安全摘要。
- 同一失败不重复堆叠多个大错误卡；按一次运行聚合展示。
- `capability_not_found` 等协议错误映射为中文产品错误和恢复动作。
- 不显示 `loom.mcp.*`、`CLI · READ` 等内部细节。

### 8.2 调试视图

调试器保留并新增：

- canonical 能力 ID 与网关别名。
- 会话模型、实际运行模型和模型来源。
- 自动范围解析结果与证据。
- 系统提示词版本、能力目录版本和工具 Schema。
- 原始错误码、脱敏输入输出、耗时和用量。

调试器是开发和深度诊断入口，不参与普通发送流程。

调试入口和视觉行为进一步约束为：

- 页头不再常驻显示调试器按钮。
- 每次运行的结果块或更多菜单提供“运行详情”，由此打开对应运行的调试视图；开发者设置可保留统一诊断入口。
- 没有运行时不显示无意义的调试入口。
- 调试面板使用完全不透明的 `surface` 背景，禁止 `bg-*/95`、玻璃模糊或正文透底。
- 窄窗口以右侧抽屉呈现时，面板和背景正文必须有清晰层级；背景可使用独立遮罩，但遮罩不能降低面板自身不透明度。
- 关闭按钮、`Escape` 和焦点返回触发入口必须可用，面板内部保持可滚动且不产生整页横向滚动。

## 9. 视觉规范

视觉方向为“安静、专业、面向重复工作”的浅色控制台：

- 不复制 Codex 深色主题，只借鉴其输入框结构、模型菜单和命令层级。
- 输入区最大宽度与对话正文一致，约 920-960 像素。
- 卡片圆角不超过 8 像素；输入区和菜单使用克制阴影而非粗边框。
- 顶部状态压缩为一行，不用“86 项能力 / 2 项授权”作为主视觉信息。
- 动态数量使用等宽数字，防止布局跳动。
- 所有按钮具备 hover、pressed、disabled、loading 和 tooltip 状态。
- 发送、附件、停止等熟悉动作使用 Lucide 图标，不使用文字方块代替。
- 1280×720 下输入区、模型菜单和发送按钮不得重叠；最长模型名必须截断并提供 title。
- 不使用透明调试面板叠压正文；信息密集面板必须采用实色表面。
- 异步操作必须有即时反馈，“正在思考”属于运行状态而非装饰动画。
- 同一命令在一个视图中只保留一个主要入口，避免页头与侧栏重复。

### 9.1 UI/UX Skill 审查结论

依据 `ui-ux-pro-max` 的可访问性、触控反馈、动效、导航和渐进披露规则，当前截图和实现存在以下问题：

| 优先级 | 问题 | 设计处理 |
|---|---|---|
| P0 | 发送后到首个响应之间没有反馈，用户无法判断是否已受理 | 增加会话内“正在思考”状态，并覆盖失败、取消、重连和减少动效 |
| P0 | 页头“新对话”与会话栏 `+` 重复 | 删除页头按钮，统一由会话栏 `+` 创建 |
| P1 | `</>` 将开发者功能暴露为普通页面主操作 | 从页头删除，调试入口下沉到具体运行详情 |
| P1 | 调试面板 `bg-surface-alt/95` 导致正文透底和视觉干扰 | 使用完全不透明表面，窄屏抽屉单独处理遮罩与焦点 |
| P1 | 技术事件与普通对话竞争视觉层级 | 普通视图聚合为中文业务动作，原始节点仅在运行详情中显示 |
| P2 | 页头能力数、授权数和运行 ID 信息密度偏高 | 页头只保留身份、连接状态和模型摘要，诊断信息下沉 |

## 10. 错误处理

| 场景 | 产品行为 |
|---|---|
| 模型被账号移除 | 阻止下一条发送，提示切回默认或打开模型管理 |
| 模型请求中途失败 | 保留会话模型，运行进入可恢复状态 |
| 工具别名无法还原 | 返回稳定协议错误，记录别名与目录版本，不尝试模糊执行 |
| 自动范围无匹配 | 在对话中说明未发现对应设备或组 |
| 自动范围多匹配 | 给出简短候选澄清，不创建写任务 |
| 设备下线 | 从最终 TaskGrant 中拒绝该设备并显示原因 |
| 能力不可用 | Agent 选择可替代能力；无替代时给出中文阻塞说明 |
| 第三方能力无中文描述 | 使用来源和清洗后的名称生成兜底显示，不暴露权限代码 |

## 11. 兼容与迁移

- 旧会话没有 `modelId` 时自动继承账号默认模型。
- 旧草稿中的 `capabilityHints` 读取后忽略，清空草稿时不再写回。
- 旧草稿中的设备和设备组迁移为 `scopeMode: manual`，避免丢失用户明确选择。
- Agent API 保持现有路径不变，只做向后兼容字段扩展。
- 已运行任务继续使用创建时的模型和范围快照，不受 UI 切换影响。
- 工具名别名映射部署后，旧会话中的历史失败保持可读，不自动重放工具。

## 12. 测试策略

### 12.1 后端

- 模型别名编码、响应还原、长度和碰撞测试。
- 会话模型继承、切换、持久化和无效模型拒绝测试。
- 运行模型快照不受后续切换影响测试。
- 全能力目录注入和中文元数据合同测试。
- 内置能力清单与 Agent 已连接目录一致性测试，禁止媒体或手机能力仅注册无执行器空壳。
- 生图、图片编辑、视频生成、单机手机任务和多机矩阵任务分别执行一次原生 Agent 工具闭环测试。
- MCP、CLI 和 Skill 动态新增、禁用、不可用及安全元数据缺失时的目录同步测试。
- 自动范围：全部在线、设备 ID、设备组、不需要、无匹配和多匹配测试。
- 自动范围不能扩大 TaskGrant 测试。
- 旧 `capabilityHints` 和旧会话兼容测试。

### 12.2 前端

- Composer 不再出现能力、设备和设备组常驻字段。
- 模型菜单读取真实账号模型并更新当前会话。
- “设为默认”只在显式点击后调用全局账号接口。
- 范围菜单使用真实矩阵设备和组，默认保持自动。
- 发送请求包含正确会话模型和范围模式。
- 技术能力名不出现在普通消息块。
- 每个按钮具备真实命令和状态，不允许点击无响应。
- `AgentHeader` 不再接收或渲染调试、新对话命令；会话栏 `+` 仍能真实创建会话。
- 发送后立即出现唯一思考状态，首个可见事件、失败、取消、停止和切换会话均能正确清理。
- 思考状态具备 `role="status"`、`aria-live="polite"` 和 `prefers-reduced-motion` 降级。
- 调试面板使用不透明背景，并可通过运行详情打开、关闭和键盘退出。

### 12.3 视觉与端到端

- Playwright 覆盖 1600×1000、1280×800、1100×720。
- 截图检查输入区、模型菜单、长模型名、附件、范围菜单和运行中停止状态。
- 真实闭环覆盖：切换会话模型 → 查询麓鸣状态 → 自动调用能力 → 返回中文结果。
- 创作闭环覆盖：自然语言要求生图、编辑图片或生成视频 → Agent 自动选择媒体能力 → 返回可打开的媒体结果。
- 单机闭环覆盖：指定一台手机读取屏幕并执行任务 → 自动调用手机能力 → 返回证据和结果。
- 多机闭环覆盖：输入“让招聘一组执行任务” → 自动解析范围 → TaskGrant → 矩阵分发。
- 模糊目标覆盖：输入“让那几台手机执行” → 只澄清，不误发任务。
- 截图验证页头无 `</>` 和重复“新对话”，调试面板打开后正文不可透入。
- 慢响应场景验证“正在思考”在首个输出前稳定可见，输出出现后无布局跳动。

## 13. 验收标准

1. 普通输入区不再展示“设备”“设备组”和“能力”三个常驻表单控件。
2. 用户可在输入区查看并切换当前对话模型，其他会话不受影响。
3. 只有显式“设为麓鸣默认模型”才修改全局默认模型。
4. 用户无需选择能力即可完成状态查询、生图、图片编辑、视频生成、单机手机控制、多机矩阵控制和其他已接入能力调用。
5. 内置能力在普通界面具备中文名称和中文描述。
6. 网关工具别名能够可靠恢复为 canonical 能力 ID，不再出现截图中的 `capability_not_found`。
7. 明确的设备、设备组和全部在线范围可以自动解析；模糊范围不会误执行。
8. 普通视图不显示内部能力 ID、英文权限代码和重复协议错误卡。
9. 页头不再显示 `</>` 和“新对话”，会话栏 `+` 是唯一常驻新建入口且可真实创建会话。
10. 发送后 100 毫秒内显示“正在思考”；首个可见事件、失败、取消、停止或切换会话时正确结束。
11. 调试器仍能通过运行详情查看完整、脱敏的模型、工具、范围和错误细节，面板背景完全不透明。
12. 模型切换、附件、范围、发送、停止、运行详情和新对话全部通过真实点击验收。
13. 现有授权、审批、硬停止、矩阵租约和急停规则不回归。
14. 受保护安装包中的原生 Agent 完成一次真实模型工具调用、一次生图、一次视频任务、一次单机手机任务和一次矩阵范围解析冒烟。

## 14. 实施分期

### 阶段 1：修复模型工具合同

- 完成工具别名编码与 canonical 还原。
- 增加碰撞、长度和原生 Agent 集成测试。
- 中文化系统提示词的基础工具规则。

### 阶段 2：会话级模型

- 扩展会话、运行、Bootstrap 和 API 类型。
- 服务端验证账号文本模型并固化运行快照。
- 前端接入模型菜单和“设为默认”命令。

### 阶段 3：原生 Composer

- 删除能力选择器和常驻目标文本框。
- 重构附件、范围、模型、发送和停止工具栏。
- 调整普通工具消息块和错误聚合显示。
- 删除页头调试和重复新对话按钮，保留会话栏唯一新建入口。
- 增加发送到首个可见事件之间的“正在思考”状态。
- 将调试入口下沉到运行详情，并改为完全不透明面板。

### 阶段 4：自动能力与自动范围

- 中文化能力元数据和系统提示词。
- 接入自动范围解析、TaskGrant 和澄清状态。
- 验证全部可用能力自动进入 Agent 工具目录。
- 连接图片、视频、单机手机和多机矩阵的真实执行器，删除只注册不执行的能力空壳。
- 建立产品能力清单与 Agent 目录一致性合同，后续新增能力默认纳入继承审查。

### 阶段 5：视觉、真实设备与安装包验收

- 完成桌面尺寸视觉检查和按钮审计。
- 完成真实模型状态查询和多机任务闭环。
- 重建受保护安装包并执行安装后 Agent 冒烟。

## 15. 明确不做

- 不复制 Codex 的深色主题、推理强度和速度选项。
- 不让会话级模型切换隐式修改全局账号配置。
- 不允许模型自由扩大设备范围。
- 不删除矩阵安全作用域，只移除暴露给普通用户的技术表单。
- 不把能力名称中文化等同于修改 canonical 协议 ID。
- 不为美化界面重写会话、矩阵、授权、安装器或账号体系。

## 16. 设计结论

麓鸣中枢智能体的正确产品形态不是让用户配置模型运行时、设备 ID 和工具列表，而是让用户描述业务目标，由麓鸣原生 Agent 在受控范围内自行选择模型工具、解析手机范围、执行任务并验证结果。模型选择保留明确的人类控制，能力和范围选择转为系统内部职责，技术细节进入调试器，普通界面只呈现可理解、可恢复、可追踪的业务过程。
