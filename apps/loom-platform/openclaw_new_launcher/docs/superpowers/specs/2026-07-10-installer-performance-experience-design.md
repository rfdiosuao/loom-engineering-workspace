# Installer Performance And Experience Design

Date: 2026-07-10
Project: 麓鸣AI矩阵获客工作台
Scope: 前置环境检测、Codex 首装、安装任务反馈和在线/完整双版本交付

## Summary

本轮优化采用双版本交付：继续提供体积较小的在线安装包，同时提供内置 Codex 组件包的完整安装包。两种安装包共用同一套快速前置检测、组件探测、校验缓存、安装进度和失败恢复逻辑。

优化重点不是缩短加载动画，而是删除不必要的工作：安装页不再运行完整诊断，不再首次进入就顺序检测四个智能体，Codex 不再下载平台包后又执行一次远程 npm 安装，组件检测不再重复启动 Appx 和 npm 子进程。检测和安装期间窗口必须可拖动，页面、侧栏和滚动保持可用，后台任务可以在用户切换页面后继续运行。

## Measured Baseline

2026-07-10 在当前开发机上的基线测量：

- 完整前置检测耗时约 14.7 至 16.1 秒，共执行 23 项检查。
- 其中 Bridge 端口段扫描约 10.8 秒，OpenClaw/ClawPanel 进程枚举合计约 2.7 秒。
- 单次 Codex 检测耗时约 10 秒。
- 单次 Codex 检测重复执行 4 次 Appx 查询、4 次 `npm prefix -g`、4 次 `npm bin -g` 和 4 次 `npm root -g`。
- 当前 Codex 清单先下载约 118 MB 的 Windows 平台包，再执行 `npm install -g @openai/codex@0.142.3`，存在第二次网络下载。
- 安装页首次打开会顺序自动检测 Codex、Claude Code、opencode 和 OpenClaw；用户点击安装时还会再次执行组件安装前检测和安装后检测。

## Goals

- 缓存存在时，安装页立即展示，不等待环境检测完成。
- 冷启动快速前置检测目标低于 2 秒，当前开发机目标低于 800 毫秒。
- 已有本地 Codex 包时，组件检测目标低于 500 毫秒且不启动 Appx/npm 子进程。
- 未安装 Codex 的外部探测最多执行一次 Appx 查询和一组 npm 路径查询。
- 在线安装只下载一次经过签名清单指定的 Codex 平台包。
- 完整安装包不依赖网络即可完成 Codex 安装。
- 下载开始后 1 秒内出现可读进度，后续持续显示阶段、百分比、已下载大小和耗时。
- 检测和安装期间窗口始终可拖动；非破坏性页面操作、侧栏导航和滚动不被全屏遮罩阻断。
- 保持组件 SHA-256 校验、签名 manifest、回滚和日志能力，不以降低安全性换速度。

## Non-Goals

- 不修改 Codex 自身登录、账号或模型协议。
- 不把完整诊断页删除；完整诊断仍用于售后和故障排查。
- 不绕过发布清单签名或组件 SHA-256 校验。
- 不把真实账号、token、授权码或客户数据写入安装缓存和性能日志。
- 不在本轮自动配置生产 CDN、DNS 或外部服务密钥；代码支持镜像 URL 顺序，生产上传单独执行并验证。

## Experience Design

### Page Entry

1. 安装页先读取本地组件状态和前置检测缓存，首屏立即可见。
2. 缓存结果带“刚刚、几分钟前、需刷新”等状态，不伪装成实时结果。
3. 后台仅运行快速前置检测，不运行完整诊断。
4. 不再自动顺序检测四个智能体。页面只做不启动子进程的轻量文件探测；用户选择组件或点击检测时才做针对性深度探测。

### Window And Navigation

- 自定义标题栏的拖动区域始终位于加载层之上，并保留 `data-tauri-drag-region`。
- 快速检测只使用角落状态提示，提示层不截获标题栏、侧栏和页面滚动的鼠标事件。
- 安装任务只锁定当前组件的安装、卸载、回滚按钮，不锁整个应用。
- 用户可在安装过程中切换页面；任务继续在 Bridge 后台运行，再回到安装页时从 Job 状态恢复进度。
- 避免透明、模糊、覆盖整页的等待框。需要阻止重复提交时，在当前组件区域显示明确状态。

### Progress Feedback

Codex 安装按这些稳定阶段展示：

1. 检查本地缓存
2. 连接下载源
3. 下载 Codex，显示百分比和 MB
4. 校验 SHA-256
5. 解压平台包
6. 验证 `codex --version`
7. 安装完成

下载源重试、镜像切换、断点续传失败转为完整下载时都写入当前任务日志，但不弹出连续重复的 Toast。成功只显示一次完成提示，失败显示可执行的下一步。

## Fast Prerequisite Architecture

新增轻量前置检测能力，与完整诊断分离：

- `diagnose_prerequisites()` 只检查安装页实际展示的 Python runtime、Node.js、npm、Git、Git Bash、uv、WebView2 和数据目录。
- 文件存在、`shutil.which` 和轻量配置检查可直接执行；不扫描 18791-18950 端口段，不枚举 OpenClaw/ClawPanel 进程，不检查手机 APK、会员网关、OpenClaw 服务快照或完整便携包完整性。
- 每项检查和总耗时写入脱敏 timing 字段，便于继续定位慢机器。
- 安装页调用新的快速接口；诊断页继续调用完整 `/api/diagnostics/run`。
- 前置修复新增 scoped 模式，只处理缺失的公共运行时和数据目录，不停止正在运行的服务、不清端口、不重建无关配置，也不重复执行两到三次完整诊断。
- 修复完成后只复检受影响的快速前置项。

缓存采用 stale-while-revalidate：先显示最近一次结果，再后台快速复检。成功和失败结果都可缓存，但 UI 必须显示采集时间；安装动作会针对当前组件所需条件做最后一次轻量确认。

## Component Detection Architecture

组件检测按“本地直接入口、已知外部入口、昂贵系统探测”的顺序执行：

1. 先检查组件安装目录和 manifest entry。
2. Codex 平台包优先识别 `package/vendor/x86_64-pc-windows-msvc/bin/codex.exe`。
3. 找到本地入口后直接执行一次 `--version`，不再查询 Appx 和全局 npm。
4. 本地入口不存在时，才执行一次 Appx 查询和一次 npm 路径快照。
5. 同一检测事务复用入口解析结果，不在可用性检查、入口解析、版本检测和启动准备中重复扫描。
6. 昂贵外部探测结果使用短期进程内缓存；显式“重新检测”可绕过缓存。

安装页不再在首次进入时顺序提交四个检测 Job。轻量探测用于快速展示，深度检测由用户动作或当前组件安装流程触发。

## Codex Installation Architecture

### Single Download

Codex Windows 平台包已经包含可直接运行的原生 `codex.exe`。安装器解压并验证这个入口后直接标记组件就绪，不再执行远程 `npm install -g`。现有签名 manifest 即使仍包含旧 install command，安装器也会在已找到且验证通过的本地 Codex 原生入口时跳过重复 npm 安装。

### Streaming Cache

- 组件下载从整包读入内存改为流式写入 `.part` 文件。
- 下载过程中增量计算 SHA-256，并按节流规则报告进度。
- 校验成功后原子移动到按组件版本和 SHA 命名的已验证缓存。
- 安装失败或用户重试时优先复用已验证缓存，不重复下载。
- 存在部分文件且服务器支持 Range 时续传；服务器忽略 Range 时安全地从零覆盖。
- 缓存校验失败立即隔离或删除，绝不继续解压。
- URL 按签名 manifest 中的顺序尝试，未来可将火山云 CDN 放在第一位、npm 官方源作为回退。

### Extraction And Verification

- 解压继续执行路径穿越保护。
- 解压后确认原生入口存在。
- 仅执行一次 `codex --version` 并校验可启动。
- 安装状态、版本和入口位置写入现有组件状态存储。
- 卸载只删除 LOOM 管理的组件目录和缓存引用，不误删用户系统中独立安装的 Codex。

## Dual Package Delivery

### Online Setup

- 保留当前约 58 MB 的核心安装包体积级别。
- 不内置 Codex 大包。
- 首次安装 Codex 时从签名 manifest 下载一次平台包，支持进度、缓存、续传和镜像回退。

### Complete Setup

- 在核心安装包中额外内置同一份经过 SHA-256 校验的 Codex 压缩包，预计安装包增加约 118 MB。
- 内置文件放入组件缓存种子目录，运行时复制或发现后仍按 manifest SHA-256 验证。
- Codex 安装直接命中本地缓存，不访问网络。
- 在线版和完整版使用相同安装代码，避免维护两套行为。

发布脚本必须明确输出两个文件名、大小、SHA-256 和类型，避免客户误下在线版。完整版构建不得写入账号、token、授权码或运行日志。

## Error Handling

- 快速前置检测失败不让页面空白，保留缓存结果并显示本次失败原因。
- 下载源超时后切换下一个 URL；重试次数有限且日志可见。
- 断点文件与 manifest SHA 不匹配时重新下载，不尝试带病安装。
- 解压或版本验证失败时保留可诊断日志并恢复旧版本。
- 用户离开安装页不取消后台 Job；重新进入后通过 Job ID 恢复。
- Bridge 重启后从持久化 Job/组件状态恢复最终结果，不能永久显示“安装中”。

## Security And Privacy

- 继续强制 manifest 签名和组件 SHA-256 校验。
- 缓存键只包含组件 ID、版本和摘要，不包含用户身份。
- 下载日志只记录源域名、字节数、耗时、重试和错误类型，不记录 URL 查询参数中的潜在秘密。
- 不执行 manifest 之外的任意命令；Codex 直接入口必须位于受管安装目录内。
- scoped 修复仍需要用户确认，但不执行与当前缺失项无关的破坏性清理。

## Verification Plan

### Automated Tests

- 快速前置检测不得调用端口段扫描、残留进程枚举、手机、会员网关或完整诊断检查。
- scoped 修复只修复缺失前置项，并且只复检一次。
- Codex 本地平台包探测不启动 Appx/npm 子进程。
- Codex 外部探测每次最多一次 Appx 查询和一组 npm 查询。
- 本地 Codex 原生入口存在时跳过 npm install command。
- 下载缓存命中时 fetcher 调用次数为零。
- 部分下载支持续传，Range 不受支持时安全回退。
- SHA 不匹配、截断文件、镜像失败和解压失败均有契约测试。
- 安装页不再自动顺序检测四组件，并保留缓存先显示、后台复检。
- BusyOverlay 不锁标题栏拖动、侧栏、滚动和非当前组件操作。

### Release Verification

- 运行 `npm run build`。
- 运行完整 Python 单元测试。
- 构建在线 NSIS 安装包和完整 NSIS 安装包。
- 在空白临时目录执行两种安装包的静默安装与 Bridge smoke test。
- 在线版验证只发生一次 Codex 平台包下载；完整版在断网条件下完成 Codex 安装。
- 使用 Computer Use 对标题栏拖动、侧栏切换、页面滚动、检测状态、下载进度、失败重试和完成状态逐项操作。

## Acceptance Criteria

- 安装页读取缓存后立即可交互，不被前置检测遮罩锁定。
- 快速前置检测冷启动小于 2 秒；当前开发机目标小于 800 毫秒。
- 本地 Codex 检测小于 500 毫秒，且外部子进程调用为零。
- 在线 Codex 首装网络下载量不超过一个 manifest 平台包，不能再出现第二次 npm 下载。
- 完整版断网可安装并运行 `codex --version`。
- 检测和安装每个阶段都可以拖动主窗口；页面文字、按钮和进度不重叠、不空白。
- 安装过程中切换页面再返回，任务状态与进度能够恢复。
- 构建、完整测试和两种安装包 smoke test 全部通过。

## Rollout Order

1. 建立性能契约和基线测试。
2. 拆分快速前置检测与 scoped 修复。
3. 优化组件入口解析和检测调用次数。
4. 实现 Codex 单次下载、直接原生入口、流式缓存和进度。
5. 调整安装页为非阻塞任务体验并保证窗口可拖动。
6. 构建在线版与完整版，执行空白环境和断网验证。
7. 在具备火山云对象存储/CDN 配置后增加首选镜像并保留 npm 回退。
