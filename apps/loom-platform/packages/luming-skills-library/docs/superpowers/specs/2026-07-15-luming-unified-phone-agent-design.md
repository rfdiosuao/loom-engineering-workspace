# 麓鸣统一手机 Agent Skill 设计

## 目标

把现有五个独立 Skill 合并为一个可安装、可触发、可持续积累任务配方的 `luming-phone-agent`。统一 Skill 必须按固定顺序完成环境自检、手机识别、任务询问、任务匹配或探索拆解、执行验证、配方同步和结果报告。

用户不再需要判断该调用场景编写、真实 App 探索、多手机监督、获客还是 BOSS 筛选 Skill。统一入口根据任务和现场状态选择内部能力。

## 范围

### 纳入统一 Skill

- 场景 Skill 编写与结构化输出。
- 真实手机 App 的安全导航和前置条件发现。
- 多手机一次派发、事件观察、轻量轮询和异常设备定点修复。
- 获客流程、草稿生成和飞书同步边界。
- BOSS 直聘简历初筛、证据评分和人工审核边界。
- 已验证任务配方的匹配、复检、更新和复用。

### 不在本次范围

- 修改 LOOM/FastAPI/Phone Matrix 产品后端。
- 绕过登录、验证码、风控、权限、会员或支付限制。
- 自动提交招聘决定、自动私信、自动评论、自动发布或自动支付。
- 把账号密码、Token、验证码、联系方式或业务隐私写入配方。
- 自动提交或推送运行时生成的 Git 改动。

## 选定架构

采用“一个总控 Skill + 内部模块 + 内置配方库”。只有 `luming-phone-agent/SKILL.md` 带有可触发入口；其余能力存放在同一 Skill 目录下的引用、脚本、Schema、示例和配方中。

不采用单文件巨型 `SKILL.md`，因为它会让运行规则、领域规则和动态配方互相干扰。不采用外部数据库作为主存储，因为首版需要离线、可打包和可审阅。

内置 BOSS 和获客配方是领域模板。未经过真实手机路线验证前，它们不能伪装成可直接运行的 `verified` 配方。

## 运行状态机

```text
SELF_CHECK
-> PHONE_DISCOVERY
-> ASK_TASK
-> RECIPE_MATCH
-> PREFLIGHT
-> REUSE | EXPLORE
-> PLAN
-> EXECUTE_VERIFY
-> SYNC_RECIPE
-> REPORT
```

状态必须顺序推进。没有完成环境和手机识别时，不得直接询问或执行手机任务；没有完成一次证据充分的执行时，不得把新配方升级为可复用状态。

## 1. 环境自检与低风险补全

自检至少覆盖：

- LOOM CLI 的真实路径、命令目录和能力清单。
- 本地 FastAPI 服务、Matrix 接口和手机 Agent 接口的可用性。
- 配方目录、Schema、索引、临时事务目录和写入权限。
- 手机连接配置、本地网络可达性和必要的非敏感配置模板。
- 源码 Skill 与安装 Skill 的同步目标是否可解析。

允许自动处理：

- 创建缺失的本地目录和空白配置模板。
- 修复非敏感路径指针和配方索引结构。
- 启动已安装、可安全启动的 LOOM 本地服务。
- 重试无副作用的状态检查和连接探测。

必须交给人工：

- 安装或卸载手机 App。
- 输入账号、密码、验证码或 2FA。
- 登录提交、系统授权、会员购买、支付或账号变更。
- 绕过平台风控、速率限制或设备安全提示。

自检失败时输出具体缺失项、影响、人工处理方式和恢复入口，不使用笼统的“环境异常”。

## 2. 手机识别与任务询问

先通过实际能力目录解析可用命令，再读取 Matrix 和 Phone 状态，不得编造命令名。

手机识别结果至少包含：

- `deviceId`、在线状态、忙闲状态和控制能力。
- 当前 App、当前页面摘要、锁屏或权限提示。
- 适合单机任务、矩阵任务或必须隔离的判断。

识别结束后再询问任务。标准提问应包含现场摘要，例如：

> 已识别到 3 台手机，其中 2 台可用。你希望手机完成什么任务？

如果用户在最初请求中已经描述任务，则改为确认标准化后的目标，而不是要求重复输入。

只有一台健康手机时默认选中。多台手机时，根据任务规模询问使用单机还是矩阵派发；任务已经明确要求多机时可直接进入矩阵规划。

## 3. 配方匹配、复检与探索

配方匹配至少使用：

- App 标识和版本范围。
- 标准化任务目标和别名。
- 入口页面、预期结果和操作模式。
- 已保存页面指纹和关键控件。

只有 `verified` 配方可进入复用候选。命中后先快速复检环境、入口页面和关键页面指纹：

- 复检通过：按已验证步骤执行。
- 页面或版本变化：标记 `stale`，转入安全探索。
- 前置条件缺失：标记本次运行 `blocked`，保留配方本身的验证状态。

新任务或过期任务使用安全导航探索。默认预算为 40 个动作或 15 分钟。同一页面指纹连续出现三次且没有新控件时，停止当前分支并报告 `exploration_budget_exhausted`。

## 4. 任务拆解与执行

任务计划必须包含：

- 目标、App、设备范围和预期结果。
- App、账号、登录、会员、权限、网络、区域、版本和目标数据等前置条件。
- 每一步的入口页面、动作、预期页面、证据和验证方式。
- 人工确认点、异常分类、恢复点和重试预算。
- 单机执行或矩阵执行策略。

多手机任务使用以下循环：

```text
preflight
-> dispatch_once
-> watch_events
-> poll_health_lightly
-> classify_each_device
-> diagnose_abnormal_only
-> repair_or_retry_target
-> summarize
```

不得因为进度不清楚而重复派发整个任务。每台手机维护独立状态、任务 ID、幂等键和重试预算；一台异常手机不能阻塞健康手机。

## 5. 安全边界

以下动作在 `safe_navigation` 中是无条件硬停止：

- 登录提交、验证码、2FA 和系统授权。
- 会员购买、支付、下单、退款。
- 发布、评论、私信、添加联系人和批量外联。
- 删除、归档、拒绝候选人、账号或资料修改。
- 任何不可逆操作或平台风险绕过。

人工确认只能批准某个明确动作，不能作为后续所有风险动作的永久授权。探索阶段可以记录动作提案和恢复点，但不得跨过硬停止继续探索。

## 6. 配方模型

每个配方至少包含：

- `schema`、`recipeId`、名称、别名和状态。
- App、版本范围、目标、入口和预期结果。
- 单机或矩阵模式及设备要求。
- 前置条件和已验证步骤。
- 页面指纹、证据摘要和验证断言。
- 风险动作、确认边界和禁止动作。
- 异常恢复、重试预算和 `resumeFrom`。
- 创建时间、最后验证时间、成功次数和来源。

状态定义：

- `draft`：已拆解但未完成一次验证，不可自动复用。
- `verified`：至少成功执行一次且每一步有证据，可进入复用候选。
- `stale`：App 版本、入口或页面指纹发生变化，必须重新探索。
- `blocked`：制作过程缺少前置条件，保留恢复点但不可复用。

## 7. 配方同步事务

安装器记录统一 Skill 的源码位置。运行时同步按以下顺序执行：

1. 在用户目录下的事务区创建脱敏配方和索引候选。
2. 获取配方库写锁，防止两个手机任务同时修改索引。
3. 使用 JSON Schema 校验配方和索引。
4. 扫描密码、Token、验证码、手机号、联系方式和个人隐私。
5. 原子写入源码 Skill 的 `recipes/` 和 `index.json`。
6. 原子写入本机安装副本。
7. 对源码和安装副本逐文件计算 SHA256 并比较。
8. 全部成功后返回 `synced`，允许 `verified` 配方参与下次匹配。

任一步失败时保留事务区副本并返回 `sync_pending`。未完成双端同步的配方不进入自动复用索引。同步不会自动提交或推送 Git。

## 8. 旧 Skill 迁移

以下旧入口被合并并停止独立安装：

- `luming-scenario-skill-writer`
- `luming-phone-scenario-builder`
- `luming-matrix-supervisor-loop`
- `luming-acquisition-agent`
- `luming-boss-resume-screening`

迁移步骤：

1. 将通用规则迁入统一 Skill 的对应内部模块。
2. 将 BOSS 和获客能力迁为内置领域配方和参考规则。
3. 将 manifest 收敛为唯一 Skill 入口。
4. 安装统一 Skill 并完成哈希验证。
5. 只删除上述五个已知旧安装目录。
6. 保留所有未受管的本机 Skill 和用户目录。

## 9. 输出契约

每次运行至少返回：

- 环境自检结果和自动修复项。
- 已识别手机、选择结果和隔离设备。
- 标准化任务目标及配方匹配结果。
- 任务计划、当前状态和人工确认点。
- 执行证据、阻塞原因或最终验证结果。
- 配方状态、同步状态、文件位置和下次是否可复用。

缺失 App、账号、登录、会员、权限或数据时，必须给出结构化阻塞报告和 `resumeFrom`。

## 10. 验收标准

实现必须覆盖以下契约测试：

1. 首次启动能补齐低风险本地环境。
2. 没有手机时准确报告连接缺口。
3. 单手机自动选择，多手机正确询问或矩阵派发。
4. 已验证配方在快速复检后复用。
5. 页面变化会把配方标记为 `stale`。
6. 新任务可安全探索、拆解、验证并写回。
7. 缺 App、账号、登录、会员或权限时返回可恢复阻塞结果。
8. 支付、发布、私信等动作停在动作前。
9. 连续三次相同页面停止探索。
10. 并发配方写入不会损坏索引。
11. 敏感数据不会进入配方。
12. 源码、安装副本和 ZIP 中的统一 Skill 文件一致。
13. 旧五个 Skill 不再触发或残留，其他 Skill 不受影响。
14. 发布包只包含一个可触发的麓鸣 Skill。

## 剩余风险

- 真实 App 页面持续变化，必须依赖每次复检和 `stale` 降级机制。
- 真实手机端到端执行需要现场设备和账号条件；本仓库能完整验证协议、Schema、同步和安装迁移，但不能伪造真实平台成功。
- 同步过程若遭遇断电或进程终止，事务区保留候选文件；下次启动必须先恢复或清理 `sync_pending`，不得静默丢失。
