# Recipe Sync Contract

Before candidate creation, validation, or privacy scan, reject any recipe that does not pass all three reuse gates: status == verified, verification.successCount >= 1, and non-empty evidence for every route step. Reject a route step whose verification is unknown or whose evidence is empty. A candidate that fails a gate is not reusable and must not be written to either index.

During recipe candidate creation, validation, and privacy scan, explicitly reject raw resumes and unrelated personal data. Also reject passwords, tokens, verification codes, phone numbers, contact information, and personal secrets. Keep only minimized task evidence required to verify an observed route.

Synchronize an eligible candidate in this order: create a sanitized candidate in a disjoint user transaction directory; acquire the recipe write lock; privacy-validate and schema-validate both existing indexes, every indexed recipe target, and any unindexed candidate target pair; build and validate the candidate index; back up affected targets; atomically write source and installed recipes before indexes; compare SHA256 for every source and installed file; durably finalize result and journal; then return synced.

Any failed stage preserves the transaction candidate and returns sync_pending. A partially synchronized recipe is not eligible for automatic reuse. Never automatically commit or push Git changes. This contract describes sync behavior only; source-install-ZIP parity enforcement is owned by later packaging work.

## Schemas And Synchronizer

Validate recipes with `schemas/recipe.schema.json` and indexes with `schemas/recipe-index.schema.json`, both strict JSON Schema Draft 2020-12 documents. In addition to schema validation, reject duplicate `recipeId` values semantically when updating the index.

Invoke `scripts/sync_recipe.py` with `--recipe-file`, `--source-skill-root`, `--installed-skill-root`, and `--state-root`. The synchronizer emits one JSON object with `schema: loom.phone-agent.recipe-sync.v1`, `status: synced|sync_pending|rejected`, source and installed recipe/index hashes, the applied redactions, and the preserved transaction path when a transaction was opened. A rejected candidate exits nonzero without modifying either target. An operational failure exits nonzero as sync_pending and restores all affected source and installed recipe/index files from the transaction backups.

The synchronizer removes unknown credential and contact keys and preserves non-secret `deviceId` and page fingerprints. Raw resumes, unrelated personal data, and private data in any schema-allowed string are rejected rather than persisted as a sanitized candidate.

Every schema-allowed string is recursively privacy-scanned. Password phrases with value-like content, auth/session/access/API/bearer/JWT tokens, alphanumeric verification codes and OTPs, unformatted call numbers, formatted/international/Chinese phone numbers, email addresses, WeChat/contact identifiers, identity-card values, resume sections, and unrelated-personal-data narratives reject the recipe. Benign UI exemptions are full-string matches for exactly `password field`, `password screen`, or `secret settings`, with only optional terminal `.`, `!`, or `?` punctuation. Evidence assertions are closed objects rather than narrative strings. Date-time fields are checked through `jsonschema.FormatChecker`.

Derive the exclusive lock from the canonical non-overlapping source/installed root pair, never from `state_root`. Canonicalize `state_root` before transaction creation and reject it when equal to, above, or below either target root or any recipe/index target. Before candidate-index persistence or backup, load and privacy-validate both existing indexes, all indexed recipes, and any source/installed candidate recipe target even when absent from the index. Candidate-target preflight checks lexical presence before file type: paired directories, broken links, other non-files, and one-sided paths are operational `sync_pending` failures. Existing file pairs must both exist, pass schema and recipe-ID checks, and be byte-identical. Indexes require byte-identical SHA256 parity, unique recipe IDs, unique paths, and the canonical path `recipes/{recipeId}/recipe.json`. Reject any write target that escapes its canonical root or crosses a symlink, junction, or reparse-point component.

Before the first replacement, persist a transaction journal in the pair coordination directory and the user transaction directory. The journal records lock ownership, phases, result status, all target paths, prior existence, backup paths, and backup SHA256 values. A later lock owner restores every incomplete dead/stale-owner journal before reading indexes or accepting new writes and persists a hash-cleared `sync_pending` result for the recovered transaction. For canonical `committed`, `rolled_back`, or `recovered` journals, recovery verifies or repairs `result.json`, mirrors the terminal phase and matching `resultStatus` into the local journal, and only then removes the canonical pointer. Rollback attempts every target, verifies restored hashes or absence, and aggregates all restoration errors. Transaction-directory and candidate-persistence failures are operational `sync_pending` results that retain recipe ID and redaction metadata. Result status and hashes remain pending/null until `result.json`, the committed journal, and canonical-journal removal all complete; any finalization failure rolls back and exits nonzero.
