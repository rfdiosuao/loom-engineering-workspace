# 产品帮助文案规范

OpenClaw 文档和界面提示应优先说明当前状态、影响范围和下一步操作。面向用户的文案以可执行、可验证、可复现为目标，避免使用无法验证的宣传性表达。

## 文案原则

| 原则 | 写法 |
| --- | --- |
| 状态明确 | “当前为预览模式” |
| 动作具体 | “点击后会打开 PowerShell 并运行 openclaw onboard” |
| 范围清楚 | “仅读取截图，不执行点击或输入” |
| 下一步可执行 | “重新保存 Token 后再验证签名” |
| 信息安全 | “密钥已脱敏显示” |

## Toast 写法

| 场景 | 标题 | 详情 |
| --- | --- | --- |
| 保存成功 | 设置已保存 | 网关、图像、视频和启动器配置已写入对应文件。 |
| 签名失败 | 同步模型失败 | Lumi 签名无效，请重新保存设备并验证安全通道。 |
| 桌面组件安装 | 桌面组件已安装 | Luminode Desktop Agent 已写入 OpenClawFiles/agents/luminode-desktop。 |
| OAuth 失败 | OpenAI OAuth 失败 | 请检查代理是否传给 OpenClaw 进程，或使用 openclaw onboard。 |
| 任务超时 | 任务超时 | 请检查锁屏、无障碍和当前截图，再拆分任务重试。 |

## 按钮写法

按钮文案应描述即将执行的动作。

| 避免使用 | 建议使用 |
| --- | --- |
| 开始托管 | 启动核心服务 |
| 完成设置 | 保存并验证 |
| 修复配置 | 重新配对 |
| 打开配置 | 打开 OpenClaw 引导终端 |
| 开始处理 | 执行任务 |

## 提示框写法

提示框建议包含三类信息：

1. 当前状态。
2. 影响范围。
3. 下一步操作。

示例：

```text
Lumi 签名无效。
当前设备基础连接可达，但安全通道没有通过校验。
请重新复制手机端 Token，保存并验证后再同步模型。
```

## 辅助提示的边界

辅助提示可以让文档更容易阅读，但不能替代操作说明。建议放在章节开头或检查清单附近，并保持一句话以内。

推荐写法：

- “首次使用建议先完成基础检查。”
- “提交交付包前，请确认用户状态已清理。”
- “排障时请保留错误原文和最近一次成功时间。”

避免写法：

- 无法验证的宣传语。
- 承诺结果但不说明条件的表达。
- 盖过操作步骤的概念化表达。

## 文档语气

1. 新手文档：减少术语，按顺序说明操作。
2. 交付文档：使用检查清单，明确清理项和验收项。
3. 二开文档：先说明边界，再给路径和命令。
4. 排障文档：先定位层级，再给证据和处理动作。
