# 麓鸣授权云完整 SaaS 后台改版设计

**状态：** 已确认设计方向，等待书面规格复核

**日期：** 2026-07-10

**范围：** `license_server/admin_console.html`、`license_server/server.py`、`license_server/tests` 及必要部署说明

**线上入口：** `https://license.heang.top/admin`

## 1. 背景

现有授权后台已经具备授权码、商家账号、邀请码、套餐、网关、销售入口、模板库和审计等能力，但页面仍以功能堆叠为主：

- 生成授权表单同时暴露套餐、功能、网关、密钥和模型等大量字段，高频发卡路径过长。
- 授权码列表、批量操作和危险的“清空全部”操作处于同一视觉层级。
- 首页缺少经营视角，管理员无法先看清授权总量、设备激活、当日发卡和服务状态。
- `GET /admin/api/codes` 当前会把所有完整授权码下发到浏览器；即使界面遮蔽，浏览器网络响应仍可读取。
- `codes.create` 的历史审计数据可能包含完整授权码，不符合授权资产默认遮蔽原则。

本次改版采用用户确认的 **完整 SaaS 后台** 方向：首页采用 **经营总览 + 快速发卡**，授权码采用 **默认遮蔽、二次确认后查看或导出、服务端审计留痕**。

## 2. 目标

1. 把后台改成适合日常商业交付的 SaaS 管理界面，而不是接口能力集合页。
2. 首页直接回答“当前经营和交付状态如何”以及“下一步要做什么”。
3. 高频发卡只显示必要字段，高级网关和模型配置回到独立模块。
4. 完整授权码不再随列表接口批量下发；查看和导出必须显式确认并留痕。
5. 保留现有 Python 单服务、单 HTML 管理端和 SQLite 部署方式，不引入新的前端构建链。
6. 不破坏现有授权激活、签名、客户端配置、安装器和发布流程。

## 3. 非目标

- 不在本轮加入支付、自动续费、发票、订单结算或第三方电商系统。
- 不把固定到期日改成“首次激活后计时”。当前授权仍使用 `YYYY-MM-DD` 固定到期日。
- 不重写 Python 服务框架，不迁移数据库，不引入 React/Vue 或外部 CDN。
- 不修改客户端授权协议、签名私钥、安装器、Gitee 上传和桌面端功能。
- 不在本轮建立独立的“批次”数据表；最近发卡记录使用真实审计记录呈现。

## 4. 信息架构

后台继续使用左侧导航，但按业务边界重新分组：

### 经营

- 经营总览

### 授权业务

- 发卡中心
- 授权码
- 激活设备
- 客户与商家
- 邀请码

### 商品配置

- 套餐与价格
- 销售渠道

### AI 能力

- 模型网关
- 模板库

### 系统治理

- 操作审计
- 系统设置
- 使用文档

超级管理员可见全部模块；商家账号只显示其被授权的页面和自有数据。前端隐藏仅用于体验，服务端 RBAC 仍是最终权限边界。

## 5. 经营总览

首页采用已确认的“经营总览 + 快速发卡”方案。

### 5.1 顶部状态

显示四个真实状态卡：

- 累计授权码：来自授权码元数据列表。
- 设备激活：汇总真实 `activations` 数量。
- 今日新建：按 `createdAt` 的本地日期统计。
- 授权服务：只有核心接口成功返回时显示“运行正常”；失败时显示“状态异常”并提供重试。

页面不得用硬编码数字。没有数据时显示 `0`，接口失败时显示“暂不可用”，不能用 `0` 伪装请求失败。

### 5.2 快速发卡

首页只保留：

- 套餐模板
- 归属商家（仅超级管理员）
- 客户或订单备注
- 生成数量
- 每码最大设备数

套餐模板决定套餐标识、功能、网关和模型配置。管理员点击生成后先进入确认层，确认层明确显示套餐、数量、设备限制、授权模式和固定到期日。确认后调用现有创建接口。

功能清单、Base URL、API Key、模型名和自定义套餐标识不在首页出现；需要覆盖模板时进入发卡中心的高级配置。

### 5.3 最近发卡活动

超级管理员首页展示最近 `codes.create` 审计记录，包括时间、操作者、数量、套餐、归属商家和备份状态。商家账号无权读取全局审计，因此改用自有授权码元数据展示最近创建记录，不显示操作者和备份路径。由于当前没有独立批次表，界面统一称为“最近发卡记录”，不虚构批次编号。

### 5.4 系统保护状态

首页底部只读显示：

- 最近一次数据库备份是否有记录
- 密钥默认遮蔽状态
- 审计日志是否可读取

这些状态来自真实接口响应或审计元数据；无法判断时显示“待检查”。商家账号不显示平台级数据库备份和全局审计状态，只显示“平台托管”。

## 6. 发卡中心

发卡中心使用三段式表单，但仍提交到现有创建接口：

1. **基础信息：** 套餐、归属、客户备注、数量、设备限制。
2. **交付策略：** 固定到期日、会员模式、功能权限；高级区域可覆盖网关和模型配置。
3. **确认生成：** 显示最终摘要、固定到期提醒、数据库备份说明和审计说明。

生成成功后：

- 完整授权码在本次结果面板中显示。
- 支持复制当前结果、下载 UTF-8 CSV 和纯文本交付文件。
- 明确提示管理员立即妥善保存；关闭结果后，重新查看需要二次确认并产生审计记录。
- 不提供“删除刚生成”作为主按钮；删除进入更多操作并二次确认。

单次生成数量继续遵守服务端现有上限，不通过前端绕过。

## 7. 授权码管理

授权码列表默认显示：

- 遮蔽后的授权码标识
- 客户和归属商家
- 套餐及权限摘要
- 已激活设备数 / 最大设备数
- 到期日
- 启用、停用、过期状态
- 创建时间

搜索支持客户、授权码尾号、套餐和归属商家。筛选支持状态、套餐和归属。列表默认不显示网关详情和完整功能清单，详情抽屉按需加载现有元数据。

批量改套餐、停用、重置设备和删除不进入顶部常驻按钮。它们位于“批量操作”或单行“更多”菜单，并在提交前显示影响数量。清空全部授权码只保留在超级管理员的系统设置危险区，要求输入确认短语。

## 8. 完整授权码安全

### 8.1 默认响应

`GET /admin/api/codes` 不再返回 `fullCode`。列表只接收 `codeHash`、`codeLabel` 和业务元数据。服务端内部读取完整码的能力与面向列表的公共序列化分离。

创建接口 `POST /admin/api/codes` 可以在成功响应中返回本次新生成的完整码，以支持一次性交付。

### 8.2 显式查看

新增 `POST /admin/api/codes/reveal`：

- 请求：`{"codeHash":"...","confirmation":"REVEAL"}`。
- 权限：已登录管理员；商家只能查看归属于自己的授权码。
- 响应：`{"code":"...","codeLabel":"..."}`，不附带网关密钥等无关秘密。
- 审计：记录 `codes.reveal`、操作者、目标尾号、IP 和时间，不记录完整码。

前端在调用前显示二次确认对话框。此处的二次确认用于防误触；真正的安全边界仍由登录会话、RBAC 和服务端归属校验提供。

### 8.3 显式导出

新增 `POST /admin/api/codes/export`：

- 请求为 `{"codeHashes":["..."],"confirmation":"EXPORT"}`，不接受隐式“导出所有”。
- 服务端限制单次导出数量，第一版上限为 500。
- 商家只能导出自有授权码；超级管理员可导出所选授权码。
- 响应返回所选完整码和交付字段，由浏览器生成 UTF-8 CSV 或 TXT。
- 审计记录 `codes.export`、数量、尾号列表、操作者、IP 和时间，不记录完整码。

reveal/export 在返回秘密前必须成功写入审计；审计写入失败时不返回完整码。CSV 导出对以 `=`, `+`, `-`, `@` 开头的文本字段做公式注入转义。

### 8.4 审计脱敏

- 新的 `codes.create` 审计只保存数量和 `codeLabels`，不保存完整 `codes` 数组。
- 审计 API 对历史 `before_json` / `after_json` 做输出脱敏，防止旧记录中的完整码或密钥进入浏览器。
- 本轮不直接改写历史审计数据库，避免破坏证据链；生产数据库仍由服务器文件权限和备份策略保护。

### 8.5 本轮安全边界

完整授权码仍以明文保存在 SQLite 中，因此拥有服务器 root 权限的人仍可读取。本轮解决的是浏览器默认批量暴露、误操作和审计接口泄漏；授权码静态加密、独立密钥管理和管理员二次身份验证属于后续安全专项，不与本次 UI 改版混做。

## 9. 数据流

### 9.1 页面加载

1. 前端读取当前账号、授权码元数据、套餐、账号和最近审计记录。
2. 各请求独立处理成功或失败，单个模块失败不能让整个首页空白。
3. 前端从真实响应计算总览指标，不依赖演示数据。
4. 商家账号不请求超级管理员专属接口。

### 9.2 发卡

1. 管理员选择模板并填写最少字段。
2. 前端展示最终摘要和固定到期日。
3. 管理员确认后调用创建接口。
4. 服务端先备份数据库，再创建授权码，再写入脱敏审计。
5. 前端显示本次完整码并允许下载交付文件。
6. 首页和列表刷新为最新真实状态。

### 9.3 查看或导出

1. 列表默认只持有遮蔽数据。
2. 管理员选择查看或勾选导出对象。
3. 前端显示数量、归属和风险确认。
4. 服务端校验会话、角色、归属和数量上限。
5. 服务端写入脱敏审计后返回最小必要秘密数据。
6. 前端使用后清除临时内存状态，不把完整码写入 LocalStorage、URL 或控制台日志。

## 10. 错误处理

- 未登录或会话失效：回到登录界面，并保留“会话已过期”提示。
- 权限不足：显示无权操作，不回显目标是否存在，避免越权枚举。
- 归属不匹配：服务端返回统一 404 或 403，不返回完整码。
- 发卡失败：不显示成功结果；服务端保留错误信息，前端提供重试但不自动重复提交。
- 导出失败：不生成空文件；保留当前选择并显示可重试提示。
- 部分首页接口失败：对应模块显示异常状态，其余模块继续可用。
- 固定到期日早于或等于当前日期：前后端都阻止生成。
- 网络重复提交：生成按钮在请求期间禁用；第一版不新增幂等键，审计记录用于核查重复生成。

## 11. 视觉与交互规范

- 延续深绿色品牌色，主内容使用白色和浅灰背景，状态色只用于成功、警告和危险。
- 卡片圆角不超过 8px；页面区块不做多层嵌套卡片。
- 顶部不放“清空全部”等危险操作。
- 按钮使用清晰命令文案；常用动作主色，次要动作描边，危险动作红色且不常驻。
- 桌面端优先，同时保证 1024px 宽度不重叠；窄屏改为单列，不允许横向遮挡关键按钮。
- 所有完整码、Token 和 API Key 默认遮蔽；复制成功不回显到 Toast。

## 12. 测试策略

### 后端单元和契约测试

- 授权码列表接口不含 `fullCode`。
- 创建接口只返回本次生成的完整码，审计中不含完整码。
- reveal/export 在未登录、越权、错误归属和超量请求时拒绝。
- reveal/export 对超级管理员和授权归属商家返回正确结果。
- reveal/export 均生成脱敏审计记录。
- 历史审计 API 输出不暴露授权码和网关密钥。
- 原有激活、套餐、账号、数据库保护测试继续通过。

### 前端契约测试

- 导航包含约定分组和页面入口。
- 首页包含真实总览、快速发卡、最近发卡和系统保护状态。
- 顶部不存在“清空授权码”。
- 授权码表格默认使用遮蔽标识。
- 查看、导出、停用、删除和清空均有对应确认层。
- 普通商家看不到超级管理员专属导航和危险区。

### 可视化验证

- 本地启动授权服务，分别用超级管理员和商家账号检查页面。
- 桌面宽屏、1024px 和窄屏检查导航、表单、表格、对话框不空白、不重叠。
- 实际走通：登录 -> 快速发 1 个测试码 -> 下载 -> 列表默认遮蔽 -> 确认查看 -> 审计出现 reveal/export。
- 使用临时数据库完成演示，不能污染生产授权数据。

## 13. 发布与回滚

1. 所有测试先在临时数据库和本地 HTTP 服务完成。
2. 部署前备份线上 `license.db`、`server.py` 和 `admin_console.html`。
3. 使用现有部署脚本上传到临时文件名，校验后原子替换，不改私钥。
4. 重启服务后检查健康状态、登录、列表遮蔽、发卡、查看、导出和审计。
5. 若验证失败，恢复上一版 `server.py` 和 `admin_console.html`；数据库只在确认有业务写入异常时使用对应备份恢复。

## 14. 完成条件

- 用户确认的完整 SaaS 信息架构和 A 版首页落地。
- 快速发卡可真实生成授权码，固定到期日提示清楚。
- 授权码列表接口和页面默认不暴露完整码。
- 查看和导出经过显式确认、服务端权限校验和审计留痕。
- 历史审计接口输出已脱敏，新的审计不再保存完整码。
- 原有授权激活、套餐、商家、网关、模板和客户端配置能力不回归。
- 本地测试和视觉检查通过后，按现有备份与部署流程更新线上后台。

## 15. 明确保留的后续债务

- SQLite 中完整授权码的静态加密和密钥轮换。
- 管理员查看秘密信息时的密码重验或 TOTP 二次身份验证。
- 授权码数量达到万级后的服务端分页、筛选和统计接口。
- “首次激活后计时”的订阅有效期模型和续费订单体系。
