# 麓鸣 2.1.56 保姆级飞书教程设计

## 目标

为第一次接触麓鸣、手机 Agent、Codex 和飞书多维表格的普通用户制作一份可以逐步照做的飞书云文档。教程必须使用 2.1.56 当前界面的真实截图，以粗红色线框、红色编号和图注明确指出每一步要点击的控件。

## 受众与成功标准

- 受众不需要理解 CLI、API、MCP、端口或 JSON。
- 每一步都包含：从哪里进入、按钮原文、要输入什么、点击后看到什么、怎样判断成功、失败怎么办。
- 用户能完成安装与授权、环境检测、AI 接入、手机连接、多设备矩阵、获客页使用、飞书表格绑定与测试写入、日志诊断。
- 不出现真实授权码、token、设备连接令牌、账号隐私、客户数据或生产表格标识。
- 不把草稿和人工确认链路描述成自动群发；真实发布、评论、私信、加好友和加微继续要求人工确认。

## 文档结构

1. 封面与“这套系统能做什么”
2. 开始前准备清单
3. 安装后第一次打开与授权激活
4. 检测并补齐运行环境
5. 让 Codex / AI 接入麓鸣
6. 连接第一台手机
7. 添加多台手机并查看矩阵状态
8. 认识获客总览
9. 飞书扫码登录
10. 新建飞书线索表或绑定已有表格
11. 测试写入与重试同步
12. 复制 AI 接入提示词并开始执行
13. 查看日志、导出诊断与处理常见故障
14. 安全边界与每日使用检查清单

## 截图规范

- 计划使用 16 到 18 张截图，分辨率以 1200 x 800 为主。
- 红框直接叠加在浏览器渲染的真实控件上，不重绘或生成 UI，避免文字失真。
- 每张图只强调 1 到 3 个动作；红框使用 4 px 实线，编号使用红底白字圆形标签。
- 图中所有账号、授权码、token、客户数据和真实表格链接必须缺省、打码或使用安全测试状态。
- 图注格式统一为“图 N：先点击 ①，再检查 ②”。

## 飞书交付方式

- 使用当前已登录的 `lark-cli` 用户身份和 `docs v2` API 创建飞书云文档。
- 先创建标题、导读和章节骨架，再分段写入正文，避免超长参数失败。
- 截图使用 `docs +media-insert` 上传；图片后紧跟对应步骤、成功判断和故障处理。
- 完成后使用 `docs +fetch --api-version v2 --detail with-ids` 回读，检查标题、章节、步骤、图片和安全提示是否齐全。

## 质量门禁

- 子智能体负责正文初稿、截图清单和漏步审查。
- 主智能体负责真实 UI 截图、红框标注、飞书写入和最终验证。
- 所有按钮名称必须能在当前源码或截图中找到。
- 所有外发动作必须写明“默认只生成草稿，人工确认后才执行”。
