# LOOM 2.3 平台模块与智能体可靠性审查

日期：2026-07-22
审查分支：`codex/agent-reliability-audit`
基线：`codex/migrate-2.3.0-update-center`

## 结论

LOOM 的平台合同、手机控制、矩阵调度、媒体素材和智能体运行时已经具备较完整的工程骨架。智能体“动不动调用失败”的主要问题并不是模型单纯不够聪明，而是能力选择、策略判定、失败恢复和错误保真四层之间存在断点。

本轮已修复八个会直接放大失败率、破坏用户判断或降低发布可信度的问题：普通内容被误判为高风险、模型选到不存在的能力后立即终止、失败能力的结构化原因没有进入检查点、干净检出缺失最新版 Agent 接入 Skill 包、Matrix 截图测试仍使用旧单机接口、更新自动检查覆盖下载状态、全屏矩阵入场动画短暂越界、Windows CI 的手机协作取消测试受 Node 冷启动竞态影响。相关回归及平台构建已通过。

目前不应宣称“所有真实业务链路绝对稳定”。模型供应商、真实账号、实体手机、平台发布和多机长时间运行仍需要一轮带外部凭据的验收与压测。

## 主要发现

### P1 已修复：内容文字触发了错误的高风险门禁

策略层此前扫描全部工具参数。普通生图提示词包含“支付”，或只读查询包含“删除账号”等字样时，也可能被归类为关键操作，继而要求一个并不存在的执行目标。

修复后只在真实可执行动作字段和写入型目标范围中分析自然语言；能力名称仍参与风险分类，手机“打开支付宝”等真实敏感动作仍受保护。

### P1 已修复：一次能力命名偏差会让整轮任务立即失败

供应商可能返回旧别名、命名空间包装名或不存在的能力名。此前运行时直接结束整轮任务，即使当前能力目录中存在正确能力。

修复后仅对 `capability_not_found` 提供一次受限重选机会，并把当前规范能力名返回模型。不会自动重试已执行的工具，也不会重放写操作；第二次选错仍然失败，避免无限循环和重复副作用。

### P1 已修复：失败原因在运行检查点中丢失

工具抛出策略或能力错误时，运行会失败，但检查点可能没有对应的 `toolResults`。界面只能显示“任务执行失败”，用户和开发者都无法判断是账号、参数、门禁、超时还是设备问题。

修复后终态检查点会保留结构化错误码、可重试性和不确定执行标记，前端已有的中文错误映射可以继续工作。

### P1 已修复：最新版 Agent 接入 Skill 包未进入干净检出

源文件、构建脚本和页面已切到 `20260721`，但 `.gitignore` 仍只允许旧 `20260720` 压缩包。开发机本地存在文件时不易发现，CI 或新电脑会直接缺包。

已通过可复现脚本重新生成 `luming-skills-library-20260721.zip`，替换旧包，并恢复 Agent 接入、获客模板和飞书集成合同。

### P1 已修复：Matrix 批量截图接口与端到端契约漂移

生产工作台已经统一使用 `POST /api/matrix/screens` 批量取屏，但端到端测试仍模拟旧的单设备截图接口。这会制造大量假失败，也会掩盖批量截图返回为空、重试和退避路径的真实回归。

测试桩、设备场景和错误重试现已全部切到批量接口，并覆盖单次取屏不触发控制写入、失败保留上一帧、聚焦状态不误发点击等行为。

### P1 已修复：后台更新检查覆盖正在下载的状态

更新中心打开超过 8 秒时，自动检查可能在下载或校验期间再次执行，把界面重置成“发现新版本”。用户会误以为点击更新没有反应，甚至重复触发下载。

自动检查现在会复用 `available/downloading/verifying/ready/restarting` 等活动会话，不再覆盖进行中的更新状态；对话框增加稳定阶段标记，端到端测试验证下载、校验、准备重启和安装移交。

### P1 已修复：全屏矩阵入场动画造成短暂底部裁切

全局页面入场动画会把直接子页面向下移动 6px。矩阵工作台是全屏固定控制面，动画过程中底部会短暂越过视口，造成控制条被裁切和视觉抖动。

手机矩阵守门层现已严格贴合应用主区域，全屏矩阵改用仅透明度淡入，不再改变几何位置；三个发布尺寸均通过边界检查。

### P1 已修复：Windows CI 手机取消测试受 Node 冷启动影响

手机子进程协作取消测试原本要求 Node 在与生产任务相同的 10 秒预算内写出启动标记。Windows CI 高负载时，Node 可能在标记写出前被测试自身的超时终止，导致同一提交一条流水线通过、另一条流水线随机失败。

测试现改用 Python 最小子进程验证取消信号和进程回收，不再把 Node 安装与冷启动性能混入被测行为；测试任务预算与启动观察窗口也已解耦。本机连续执行 10 次通过，手机路由测试 92 项通过。

### P2 待优化：每轮都向模型发送完整能力目录

当前约 79 项可用能力、序列化后约 44 KB，每轮模型调用都会携带全部工具。能力越多，模型越容易发生近义能力混淆，首字延迟和上下文成本也会增加。

建议下一阶段实现“两级能力路由”：先按意图和请求范围召回小目录，再保留一次完整目录兜底。该优化必须保证跨域任务仍能连续执行，不能通过简单隐藏能力来换取表面成功率。

### P2 待优化：供应商与外部账号仍是主要真实失败源

本机已保存的少量真实失败样本包含 `AGENT_ACCOUNT_LOGIN_REQUIRED` 和 `media_job_failed`。工程合同能够覆盖错误展示，但不能替代真实模型账号、图像供应商、发布账号和网络波动验收。

建议把账号过期、HTTP 401/429/524、模型无工具调用、媒体生成超时分别纳入发布前冒烟测试，并保留一键重试或切换模型入口。

### P2 待优化：审计日志无限追加

Bridge 运行日志已有大小限制和轮转；`loom-action-trace.jsonl`、`loom-task-ledger.jsonl`、`mcp-audit.jsonl` 仍由 `append_audit_record` 直接追加。长期运行会增加诊断、导出和磁盘占用成本。

建议单独迁移为带归档数量的 JSONL 轮转，并验证 CLI 查询跨归档读取，避免在本轮智能体修复中同时改变审计语义。

## 智能体可靠性发布门槛

1. 规范能力首次命中率达到 95% 以上；一次重选后达到 99% 以上。
2. 同一只读能力在一轮中不重复执行，除非上次结果明确可重试。
3. 所有失败必须落入账号、参数、门禁、网络、超时、设备、供应商或内部错误之一，禁止只返回“任务失败”。
4. 写操作、发布、支付和删除继续保留目标约束与审计；能力重选不得绕过门禁。
5. 多机任务的“已受理”和“已完成”分开展示，超时不得把手机仍在执行误报为确定失败。
6. 每个发行版至少执行一次生图、传手机、手机状态、单机任务、矩阵任务、急停和更新检查冒烟测试。

## 后续优先级

1. 建立真实模型与实体手机的发行冒烟环境，先把失败分类和成功率量化。
2. 设计两级能力路由，减少每轮工具数量，同时保留完整目录兜底。
3. 增加只读工具结果复用，避免模型重复检查系统状态和手机状态。
4. 为 Matrix 定义下发响应、急停响应、首屏截图和批量截图的延迟指标。
5. 独立迁移审计日志轮转，并保持跨归档查询和导出兼容。
