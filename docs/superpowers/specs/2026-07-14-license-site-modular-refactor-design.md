# 麓鸣授权站模块化重构设计

**状态：** 已确认方向，等待书面规格复核

**日期：** 2026-07-14

**线上入口：** `https://license.heang.top/`、`https://license.heang.top/admin`

**范围：** `license_server` 服务端、管理后台、公开授权首页、测试与授权站部署脚本

## 1. 背景

麓鸣授权站已经承担真实商业授权、设备激活、客户与商家账号、邀请码、套餐、模型渠道、发卡入口、模板库、审计和发布中继等职责。现有实现能够稳定运行，但主要逻辑集中在约 4800 行的 `server.py` 和约 3300 行的 `admin_console.html` 中。

继续在两个大文件中叠加功能会带来以下风险：

- HTTP 路由、数据库访问、鉴权、业务规则和序列化互相耦合，局部修改容易影响无关接口。
- 管理后台状态、API 调用、页面渲染和弹窗逻辑混在同一脚本中，难以单独验证。
- 发布时需要整体替换大文件，无法准确判断一个改动影响了哪些业务域。
- 单元测试主要集中在完整流程，定位失败根因和构造边界场景的成本持续增加。
- 商业 UI 已经具备较多功能，但导航和操作层级仍受单文件结构限制。

本次采用用户确认的 **“轻量部署不变，后端模块化与后台 UI 同步整理”** 方案。重构目标是降低长期维护风险，不是更换技术栈或重写业务。

## 2. 目标

1. 保留现有 Python 单服务、SQLite、Ed25519 签名和静态前端部署方式。
2. 将配置、数据库、认证、授权、客户、套餐、模板、审计和发布中继拆成边界清晰的模块。
3. 将管理后台拆成可独立理解和测试的页面模块、公共状态和 API 层。
4. 同步整理后台信息架构，让高频经营与发卡操作更直接，低频治理能力保持可达。
5. 保持 LOOM 客户端现有激活、会员、公开配置和发卡网站配置协议兼容。
6. 建立可分阶段验证、可快速回滚且不覆盖生产数据的发布流程。

## 3. 非目标

- 不迁移生产 SQLite 数据库。
- 不更换或重新生成生产 Ed25519 私钥。
- 不迁移到 FastAPI、Django、React、Vue 或其他框架。
- 不加入支付、自动续费、发票、订单结算或第三方支付回调。
- 不修改 LOOM 安装器、Gitee 上传、客户端打包或授权签名协议。
- 不在结构重构阶段重新解释现有套餐、功能权限或设备激活规则。
- 不把多个服务拆成微服务，不引入消息队列、Redis 或外部数据库。

## 4. 总体方案

采用 **模块化单体**：一个 Python 进程、一个 SQLite 数据库、一个 systemd 服务，对外继续暴露同一组 URL。

`server.py` 变为兼容启动入口，只负责解析命令、加载配置和启动 HTTP 服务。业务实现放入 `luming_license` 包，管理前端放入 `web` 目录。外部调用者不需要知道内部目录变化。

目标目录结构：

```text

  luming_license/
    config.py

    db.py

    http/
```

模块之间只能通过公开函数和明确数据结构通信。HTTP 路由不能直接拼接 SQL；领域模块不能直接写 HTTP 响应；前端页面模块不能自行拼接认证头或持久化会话秘密。

## 5. 服务端边界

### 5.1 配置与启动

`config.py` 统一读取环境变量、路径、端口、CORS、会话有效期、网关默认值和发布中继配置。配置在进程启动时解析一次，非法整数和缺失路径产生明确日志。

`app.py` 负责初始化数据库、默认模板、路由表和 `ThreadingHTTPServer`。`server.py` 保留现有 `serve`、`create-code`、`list-codes`、`public-key` 命令，内部转调新模块，避免运维命令失效。

### 5.2 数据库

`db.py` 统一提供连接、事务、初始化和在线备份能力。业务模块通过短事务访问 SQLite，不共享跨请求连接。

第一阶段保持现有表结构和字段含义不变。任何必要的兼容初始化必须是幂等操作；启动时不得删除、重建或批量改写生产表。

### 5.3 认证与安全

`security.py` 负责密码哈希、会话 Token、`HttpOnly Cookie`、CORS、安全响应头、速率限制和秘密脱敏。

现有 Bearer、`X-Admin-Session` 和 `X-Admin-Token` 兼容入口继续保留。浏览器管理后台默认只使用同源安全 Cookie，不把会话 Token 写入 LocalStorage、URL 或日志。

### 5.4 业务领域

- `accounts.py`：账号、角色、状态、商家归属和邀请码注册。
- `sessions.py`：管理会话创建、校验、吊销和过期处理。
- `licenses.py`：授权码生成、列表、遮蔽、查看、导出、启停、批量修改和删除。
- `activations.py`：设备激活、设备归属、激活次数和解绑。
- `plans.py`：套餐模板、功能、有效期和模型渠道继承。
- `public_settings.py`：发卡网站、支持入口和 LOOM 客户端公开配置。
- `templates.py`：AI 内容模板管理和公开读取。
- `audit.py`：脱敏审计、变更快照和备份路径记录。
- `publish_relay.py`：现有任务提交、租约、轮询、确认和失败重试。

领域模块保留现有错误状态码和中文错误语义。跨租户对象仍统一返回不可区分的未找到结果，避免通过错误差异探测他人数据。

### 5.5 HTTP 路由

路由按公开接口、认证接口、管理接口和发布中继分组。路由层只执行以下动作：

1. 解析和校验输入。
2. 取得当前认证上下文。
3. 调用领域函数。
4. 生成统一响应和安全响应头。

现有 URL、HTTP 方法、成功状态码和主要 JSON 字段保持不变。新增内部路由表不得改变 Cloudflare、Nginx 或 LOOM 客户端的调用方式。

## 6. 管理后台信息架构

后台继续使用安静、密集的 B2B 控制台布局，不改成营销落地页。

### 6.1 经营总览

- 授权码总数、有效授权、设备激活、今日发卡。
- 授权服务、数据库备份、公开配置和发布中继状态。
- 快速发卡入口和最近操作记录。

所有数字来自真实接口。请求失败显示“暂不可用”，不得用 `0` 伪装失败。

### 6.2 授权业务

- 授权码：搜索、筛选、查看、导出、启停和受保护的危险操作。
- 激活设备：设备与授权关系、最近激活和解绑。
- 客户与商家：账号、归属、授权数量和状态。

### 6.3 商品与销售

- 套餐配置：有效期、设备额度、功能权限和模型配置继承。
- 发卡网站配置：继续控制 LOOM 中的购买授权入口和公开授权首页购买按钮。

### 6.4 AI 能力

- 模型渠道：Base URL、模型名和密钥状态，秘密默认遮蔽。
- 模板库：图像、视频和营销模板的启停、排序与编辑。

### 6.5 系统治理

- 管理账号与邀请码。
- 操作审计与备份记录。
- 服务设置、使用文档和诊断状态。

超级管理员可见全部模块；商家只看自有授权、模型配置和必要文档。前端可见性只改善体验，服务端 RBAC 始终是最终边界。

## 7. 前端结构

前端使用原生 ES Modules 和 CSS，不增加 Node 构建依赖。`/admin` 继续是稳定入口，Python 服务负责返回 HTML 和 `/admin/assets/*` 静态资源。

- `api.js`：统一 `fetch`、同源 Cookie、错误解析和会话过期处理。
- `state.js`：保存当前账号、页面和非秘密业务数据，不做永久会话存储。
- `dialogs.js`：通用确认、危险操作、结果和编辑对话框。
- `pages/*.js`：每个业务页面独立加载、渲染和绑定事件。
- `app.js`：启动、导航、角色可见性和页面生命周期。

页面模块失败不能导致整个后台空白。每个页面拥有加载、空数据、错误和就绪状态。危险操作继续要求输入确认词，并由服务端再次校验。

## 8. 数据流

### 8.1 登录

1. 浏览器提交账号和密码。
2. 服务端执行速率限制、密码校验并创建数据库会话。
3. 服务端写入 `Secure`、`HttpOnly`、`SameSite=Strict` Cookie。
4. 浏览器通过 `/admin/api/me` 恢复账号状态。
5. 退出时服务端吊销会话并清除 Cookie。

### 8.2 发卡

1. 管理员选择套餐和归属，填写最小必要信息。
2. 前端显示数量、到期日、设备额度和能力摘要。
3. 服务端校验权限并创建数据库在线备份。
4. 服务端生成授权码、写入脱敏审计并返回本次结果。
5. 前端允许复制或导出本次结果，不进行永久浏览器存储。

### 8.3 公开配置

1. 管理员在发卡网站配置页保存购买与支持入口。
2. 服务端写入 `public_settings` 并保留审计和备份。
3. LOOM 通过现有 `/api/client/config` 动态读取。
4. 授权公开首页使用同一配置渲染购买与支持按钮。

### 8.4 激活

现有客户端激活、会员查询、用量和签名响应保持原样。重构后的路由只负责转调领域函数，签名输入、设备限制、到期逻辑和错误码不得变化。

## 9. 错误处理与可观测性

- 业务错误使用统一异常类型，携带状态码、稳定错误码和可读中文消息。
- 未捕获异常返回最小化 `500` 响应，完整堆栈只进入服务器日志。
- 请求日志不得记录密码、授权完整码、API Key、Cookie 或会话 Token。
- 数据库写操作失败必须回滚事务；审计或备份失败时，敏感查看和危险变更不得继续。
- 管理页面按模块显示错误并提供重试，不因为一个统计接口失败而隐藏其他功能。
- 健康检查继续支持 `GET` 和 `HEAD`，发布脚本同时检查服务、页面标识和关键响应头。

## 10. 兼容策略

重构前先生成以下行为快照：

- 所有公开、管理和中继路由的 URL 与 HTTP 方法。
- 典型成功响应的字段集合。
- 未登录、权限不足、跨租户、无效输入和不存在对象的状态码。
- LOOM 使用的公开配置、激活、会员和发布中继契约。
- 数据库表、索引和生产数据聚合数量。

模块拆分阶段以这些快照为不可破坏契约。内部函数可以变化，外部行为只有在规格明确批准并增加迁移说明后才能变化。

## 11. 实施阶段

### 阶段一：安全网

- 补齐路由清单、响应契约、数据库结构和静态资源测试。
- 保留当前实现作为兼容基线。
- 建立旧实现与新模块的同输入对照测试。

### 阶段二：后端模块化

- 先抽配置、响应、安全和数据库，再按领域迁移业务。
- 每迁移一个领域即运行其单元测试、完整授权流程和兼容快照。
- `server.py` 在迁移期间可以转发到新模块，但不得同时维护两套业务规则。

### 阶段三：前端组件化与 UI 整理

- 先抽 API、状态和对话框，再迁移各业务页面。
- 总览和发卡优先，随后迁移授权、客户、套餐、AI 能力和治理页面。
- 每个页面迁移后执行桌面和窄屏可视化检查。

### 阶段四：发布结构

服务器采用以下布局：

```text
/opt/openclaw-license/
  license.db
  private_key.b64
  admin_token.txt
  openclaw-license.env
  releases/<timestamp>/
  current -> releases/<timestamp>
  backups/
```

数据库、私钥、令牌和环境文件保留在共享目录，不进入版本发布目录。新版本先上传到独立 release 目录，远端编译并使用生产数据库的只读副本启动检查，通过后再切换 `current` 符号链接并重启服务。

健康检查失败时只回滚 `current`，绝不自动恢复旧数据库。数据库恢复属于独立的破坏性操作，必须再次明确确认。

## 12. 测试策略

### 12.1 自动测试

- 保持现有授权服务全量测试全部通过。
- 为每个领域模块增加独立单元测试。
- 增加所有 HTTP 路由的兼容契约测试。
- 增加 Cookie、CORS、CSP、HSTS、速率限制和跨租户对抗测试。
- 增加数据库在线备份、启动幂等和旧数据库直接启动测试。
- 增加前端模块存在性、无秘密持久化和危险操作确认契约测试。

### 12.2 可视化测试

- 桌面视口检查总览、发卡、授权列表、客户、销售配置和模型配置。
- 窄屏检查导航、表格横向滚动、弹窗固定定位和按钮不重叠。
- 验证未登录、空数据、接口失败、会话过期和权限不足状态。
- 不在生产环境点击删除、解绑、清空或生成授权等写操作。

### 12.3 发布验证

- 本地全量测试通过。
- 远端 `py_compile` 和静态资源清单通过。
- 生产数据库完成 SQLite 在线备份。
- 切换后验证 `/health`、`/`、`/admin`、公开配置和未登录 `401`。
- 比较部署前后授权码、账号、激活和套餐的聚合数量。
- 检查服务错误日志并记录备份与回滚路径。

## 13. 完成条件

1. `server.py` 只保留兼容命令和启动逻辑，目标不超过 300 行。
2. 单个后端领域模块目标不超过 800 行，单个前端页面模块目标不超过 500 行。
3. HTTP 与数据库兼容快照全部通过，LOOM 无需重新打包即可继续激活和读取发卡配置。
4. 后台总览、发卡、授权、客户、套餐、销售、AI 和治理页面均可运行。
5. 页面没有演示数字、空白主状态、不可见弹窗或不受保护的危险操作。
6. 生产部署保留完整数据库、私钥、环境配置和可用回滚版本。
7. 发布后服务正常、聚合数据一致且错误日志为空。

## 14. 风险与控制

- **模块拆分遗漏隐式依赖：** 先写行为快照，按领域逐步迁移，禁止一次性搬空大文件。
- **数据库初始化改变生产结构：** 第一阶段冻结 schema，使用生产副本做启动验证。
- **前端模块加载失败导致白屏：** 保留最小 HTML 错误壳，每页独立错误边界并增加静态资源测试。
- **Cookie 或 CORS 回归：** 保留请求头兼容入口并做同源、允许来源和恶意来源三类测试。
- **发布多文件不一致：** 使用完整 release 目录和原子符号链接切换，不逐个覆盖线上文件。
- **重构期间继续新增业务：** 新需求先进入独立规格；除紧急生产修复外，不在模块迁移提交中混入新业务规则。

## 15. 最终决策

本次重构采用以下固定原则：

- 选择模块化单体，不选择全面框架化或微服务化。
- 后端结构与后台 UI 同步整理，但按阶段提交和验证。
- 外部协议、生产 SQLite、签名私钥和 LOOM 配置链路保持兼容。
- 先建立测试安全网，再拆后端，再拆前端，最后调整发布布局。
- 每一阶段都必须可独立回滚，不以生产数据恢复作为普通回滚手段。
