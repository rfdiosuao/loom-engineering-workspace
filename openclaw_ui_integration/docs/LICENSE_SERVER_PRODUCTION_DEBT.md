# 授权后台生产级技术债台账

> 范围：`license_server` 授权码后台、会员月卡、网关配置、套餐配置、运维上线。  
> 当前状态：已经具备基础 SaaS 控制台、单码编辑、批量改月卡、会员刷新重签、操作审计、变更前自动备份能力。  
> 目标：从“能用”推进到“销售、售后、运营可以稳定使用”。

## P0：上线前必须处理

### P0-1 操作审计缺失

证据：后台可以修改授权码、网关 Token、套餐和配额，但没有记录谁改了什么。

影响：生产环境出问题后无法追责，也无法判断是误操作还是系统故障。

状态：已补。`license_server` 会记录创建、编辑、批量修改、启用/停用、删除、清空等后台操作；审计记录会脱敏网关 Token，并显示在后台“操作审计”表格中。

修复建议：

- 新增 `audit_logs` 表。
- 记录操作人、操作类型、目标授权码、修改前、修改后、时间、IP。
- 后台新增“操作审计”视图。

验收方式：

- 修改授权码后，后台能看到一条操作日志。
- 日志里能看到修改前后字段。

### P0-2 数据库备份和回滚缺失

证据：后台已经支持清空全部和批量修改，但没有一键备份/回滚流程。

影响：误操作会直接影响生产授权。

状态：已补第一阶段。所有后台数据变更前都会自动复制一份 `license.db` 到 `LICENSE_BACKUP_DIR`，默认是 `/opt/openclaw-license/backups`；审计记录里会显示对应备份文件名。

修复建议：

- 每次批量修改前自动备份 `license.db`。
- 清空全部前强制二次确认。
- 文档中固定回滚命令。

验收方式：

- 批量操作前生成 `license.db.bak-YYYYMMDDHHMMSS`。
- 能用备份文件恢复。

### P0-3 Token 管理仍然偏粗

证据：当前是单管理员 Token。

影响：销售、售后、开发共用一个 Token 时，无法区分操作者。

状态：短期止血已补。后台审计会记录管理员 Token 的哈希短标识，文档固定了 Token 轮换方式；多管理员/RBAC 仍归入 P2 治理债。

修复建议：

- 短期：继续单 Token，但必须定期轮换。
- 中期：支持多管理员和角色。

验收方式：

- 至少有明确的 Token 更换文档。
- 生产服务器不把 Token 写进仓库。

## P1：生产体验债

### P1-1 套餐模板缺失

证据：月卡、季卡、年卡仍然靠人工填写。

影响：销售容易填错模型、配额、网关。

状态：已补第一阶段。后台新增套餐模板，默认包含月卡、季卡、年卡、VIP 月卡；管理员可以新增/编辑/停用模板，新建授权和批量修改时可以套用模板。

修复建议：

- 新增 `plans` 表。
- 后台支持套餐模板：月卡、季卡、年卡、VIP 月卡。
- 创建授权码时直接选择套餐。

验收方式：

- 选择“月卡”后自动填入到期天数、配额、模型和权限。

### P1-2 搜索筛选分页缺失

证据：授权码列表目前是一次性加载。

影响：码多以后查找客户困难，页面会变慢。

状态：已补第一阶段。后台支持按客户名、完整码、尾号搜索，支持启用/停用、已激活/未激活、会员/普通筛选，并按 50 条分页展示。

修复建议：

- 增加客户名搜索。
- 增加状态筛选：启用/停用/会员/普通/已激活/未激活。
- 增加分页。

验收方式：

- 可以按客户名定位授权码。
- 1000 条授权码时后台仍然可用。

### P1-3 客户激活详情不够直观

证据：列表只显示激活次数，不显示具体设备。

影响：售后无法判断客户到底在哪台机器激活。

状态：已补。授权码列表可打开激活详情，查看 `installId`、`deviceId`、激活时间、套餐，并支持删除单条激活记录作为售后解绑。

修复建议：

- 授权码详情里展示 `installId`、`deviceId`、激活时间。
- 支持单独删除某次激活，作为售后解绑。

验收方式：

- 售后可以对单个客户解绑设备。

## P2：长期增强债

### P2-1 配额扣量未闭环

证据：后台能下发配额 JSON，但没有真正扣量。

影响：月卡只是权限配置，不是完整计费系统。

修复建议：

- 增加使用量上报接口。
- 按天/月统计 image/video/task 消耗。
- 超额后返回明确错误。

验收方式：

- 图片额度用完后，客户端拿到“额度不足”错误。

### P2-2 网关 Token 缺少密文保护

证据：当前 Token 存在 SQLite 字段里。

影响：服务器被读库时，Token 泄露风险较高。

修复建议：

- 至少做到后台不回显 Token。
- 中期做服务端加密存储。

验收方式：

- 后台列表只显示 Token 已配置。
- 编辑时留空表示不改 Token。

### P2-3 多租户和角色权限未做

证据：后台没有销售、售后、运营角色。

影响：团队使用时权限边界不清楚。

修复建议：

- 管理员可以创建子账号。
- 销售只能生成码。
- 售后只能查询和解绑。
- 管理员才能改网关和清空数据。

验收方式：

- 不同角色登录看到的按钮不同。

## 当前建议执行顺序

1. 配额扣量。
2. 多管理员/RBAC。
3. 网关 Token 加密存储。
4. 更细的审计筛选和导出。
5. 套餐模板批量导入/导出。

## 暂不建议做

- 暂不做完整支付系统。
- 暂不做复杂 CRM。
- 暂不做多租户大后台。

先把授权、月卡、售后解绑、审计、备份这条主线做稳。
