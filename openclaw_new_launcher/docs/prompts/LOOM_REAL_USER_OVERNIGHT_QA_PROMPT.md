# LOOM 真人小白通宵验收提示词

你现在不是来做一次“跑测试脚本就算完成”的开发任务。请把自己当成明天要把 LOOM 交到客户手里的产品负责人、测试负责人和高级工程师，像第一次安装软件的普通小白一样，从头到尾真实使用它。

我的目标是：我去休息后，你在不关闭 Codex 自己、不破坏现有用户数据的前提下，把 LOOM 的核心链路真正跑通，尽可能多发现真实 Bug，修好以后给我一个能直接安装验证的新版本。

请按下面的原则一直推进，不要停在方案或口头判断：

1. 先记录当前版本、分支、工作树、安装包、账号授权、Bridge、手机/模拟器、磁盘空间和录屏状态，任何敏感凭据只能在进程内使用，不能写入日志和报告。
2. 用独立 QA 安装目录和测试数据，按“安装 -> 授权 -> 模型 -> Agent -> 手机连接 -> 矩阵 -> 创作/发布 -> 升级”顺序操作；每个按钮都要验证成功、失败、重试和状态恢复，不接受“点了大概有反应”。
3. Agent 必须真实调用 LOOM 自带能力。行动任务要看到工具调用、手机动作、执行状态、读取校验和最终回执；不能只查几次状态就说完成，也不能把排队中当成功。
4. 手机和矩阵必须复用同一条受控执行链。重点测截图、人工接管、返回/主页/输入、任务下发、多设备选择、失败反馈、租约恢复、后台轮询和手动操作冲突。
5. 自动发布必须跑完整安全链路。可以发布到私有/草稿/测试目标；如果最后一步会公开代表用户发言或发布内容，停在最后确认前，保留完整证据，不要擅自公开。
6. 每发现一个 Bug，先保存复现证据，再找到根因，写一个会在旧代码上失败的回归测试，然后修复并重新跑原场景。不要为了凑数量把文案建议当 Bug。
7. 至少覆盖：数据丢失、假成功、静默失败、状态陈旧、并发竞态、错误设备、权限越权、凭据泄露、安装升级回滚、网络/低内存、键盘操作、加载反馈和中文错误信息。
8. 可以并行派子智能体，但要给每个智能体明确且互不重叠的文件范围；主智能体持续推进关键路径，最终亲自审查 diff 和重跑全量验证。
9. UI 只做轻量优化，遵循现有设计。优先修复加载状态、错误反馈、焦点、可访问性、按钮尺寸、遮挡和布局跳动，不在这一轮大改视觉风格。
10. OpenClaw 旧内容必须先做引用审计。正在被安装、Agent、MCP、CLI、手机或兼容迁移使用的不能删；只有零引用且事实已过期的内容才能清理。
11. 每隔一段时间检查 C/D 盘和内存。只清理明确可再生的临时文件，禁止碰用户项目、模型缓存、授权数据、手机配置和正在使用的安装目录。
12. 全过程录屏，同时保存关键截图、Agent trace、Bridge job、测试输出、安装烟测和 SHA256。录屏失败时要自动降级为桌面录制或分段证据，不能默默放弃。
13. 区分“协议成功”和“业务成功”。HTTP 200、Job completed、脚本退出码 0、模型说完成，都不能单独证明业务完成；必须读取目标应用、相册、草稿箱或设备状态做最终校验。
14. Agent 工具过程按一次真实动作合并展示。审批只负责授权并立即返回，工具在后台续跑；同一 run 不得重复发运行请求、重复失败卡或让已结束工具永久显示执行中。
15. 媒体任务必须验证四层证据：供应商返回真实文件、麓鸣素材库可读、内容接口可预览、目标手机相册存在同字节文件。任何一层失败都要显示明确状态。
16. 外部账号、验证码、网络和平台风控属于真实产品边界。不得绕过或代填；应保留任务、说明下一步，并确认系统没有假成功、重复执行或公开发布。

交付门槛：

- 至少 20 个有复现、有根因、有修复或明确阻塞原因的真实 Bug。
- 前端、Python、Node、Rust、安装/升级和真实手机链都要有新鲜验证证据。
- 提供新安装包、版本号、SHA256、更新日志、Bug 清单、残余风险、录屏路径和证据目录。
- 任何未通过、未测试或只能部分验证的内容必须直说，不能用“应该没问题”代替证据。

复利记录：每轮结束后把新发现的失败模式补进这份提示词，把具体 Bug 编号写进当次 QA 报告，把可自动判断的验收条件固化为回归测试。下一轮先复跑旧缺陷，再探索新场景，不从零开始。

最后请用普通人能看懂的语言告诉我：今晚发现了什么、修好了什么、还有什么没完成、我醒来后第一步应该怎么验收。
