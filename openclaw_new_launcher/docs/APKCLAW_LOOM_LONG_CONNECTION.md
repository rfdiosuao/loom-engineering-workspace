# APKClaw <-> LOOM Long Connection

Date: 2026-07-02

This document describes the first long-connection path between the APKClaw phone app and the LOOM desktop bridge. The channel is read-only and event-oriented: phone status, runtime metrics, and recent task snapshots are pushed to LOOM, while actions still use the existing signed HTTP fast path.

## Design Boundary

- Transport: Server-Sent Events over HTTP (`text/event-stream`).
- Direction: phone -> computer.
- Security: same Lumi HMAC signed channel as `/api/lumi/*`.
- Action execution: unchanged; `observe_fast`, `action_fast`, `template_run`, and Agent tasks still use existing signed request routes.
- UI impact: no LOOM main UI or installer refactor.

SSE was chosen because the channel is one-way status push. It avoids repeated polling without introducing a bidirectional command socket.

## Phone Endpoint

`GET /api/lumi/events`

Required headers are generated by the existing Lumi signer:

- `X-LUMI-LAUNCHER-ID`
- `X-LUMI-TIMESTAMP`
- `X-LUMI-NONCE`
- `X-LUMI-BODY-SHA256`
- `X-LUMI-SIGNATURE`
- existing APKClaw phone token headers are still sent by LOOM helpers for compatibility

Response headers:

- `Content-Type: text/event-stream; charset=utf-8`
- `Cache-Control: no-cache`
- `Connection: keep-alive`
- `X-Accel-Buffering: no`

### Events

`hello`

```text
id: 1
retry: 3000
event: hello
data: {"schema":"apkclaw.event_stream.hello.v1","timestampMs":1783000000000,"version":"6.38-stability","snapshotIntervalMs":1000}
```

`snapshot`

```text
id: 2
event: snapshot
data: {"schema":"apkclaw.event_stream.snapshot.v1","timestampMs":1783000001000,"version":"6.38-stability","status":{},"metrics":{},"tasks":[]}
```

Snapshot fields:

- `status`: same core compatibility fields as `/api/device/status`, plus best-effort `currentPackage`.
- `metrics`: runtime speed and queue snapshot from APKClaw.
- `tasks`: up to 5 recent async task public snapshots. Raw prompt, raw result, raw events, and tool parameters are not streamed.
- `tasks[].agentReport`: one-glance task report with `headline`, `currentStep`, `completed`, `completedSummary`, `needsCodex`, and optional `exception` / `fixHint`.

Phone stream limits:

- Snapshot interval: 1000ms.
- Active streams: max 2 per APKClaw process.
- Stream lifetime: max 1 hour per connection, then the LOOM subscriber reconnects.

`agentReport` example:

```json
{
  "schema": "apkclaw.agent_report.v1",
  "status": "error",
  "headline": "exception: Accessibility service is not running",
  "currentStep": "failed",
  "completed": true,
  "completedSummary": "Task stopped before completion",
  "needsCodex": true,
  "exception": {
    "code": "accessibility_off",
    "repairTarget": "apkclaw_accessibility",
    "codexInstruction": "Inspect accessibility binding, stale enabled state, keep-alive, and background kill recovery."
  },
  "fixHint": "Inspect accessibility binding, stale enabled state, keep-alive, and background kill recovery."
}
```

## LOOM Desktop Bridge

Node consumer:

```powershell
node scripts\openclaw-phone-agent.mjs events-sync --json --max-sec 3600 --max-events 0
```

The command prints one NDJSON object per event:

```json
{"ok":true,"type":"phone_event","deviceId":"phone-1","receivedAt":"2026-07-02T14:00:00.000Z","id":"2","event":"snapshot","data":{"schema":"apkclaw.event_stream.snapshot.v1"}}
```

At the end of a subscription window it also prints:

```json
{"ok":true,"type":"phone_event_sync_summary","deviceId":"phone-1","eventCount":12,"elapsedMs":3600000,"stoppedBy":"max_sec"}
```

Python bridge routes:

- `POST /api/phone/events/start`
- `GET /api/phone/events/status?deviceId=phone-1`
- `POST /api/phone/events/stop`

Start body:

```json
{
  "deviceId": "phone-1",
  "maxSec": 3600,
  "maxEvents": 0
}
```

CLI wrappers:

```powershell
python python\loom_cli.py phone events-start --max-sec 3600 --json --permission read
python python\loom_cli.py phone events-status --json --permission read
python python\loom_cli.py phone events-stop --json --permission read
```

Matrix sync:

- `snapshot` events update `MatrixControlPlane.register_device`.
- Presence summary includes foreground package/app, accessibility state, screen state, lock state, running-task count, and stream latency marker.
- Presence patch also includes structured fields for command brains: `busy`, `currentTaskId`, `currentStep`, `headline`, and `needsCodex`.
- The bridge keeps only a bounded recent event window to avoid long-running memory growth.

## Failure Behavior

- Missing phone URL: Node exits quickly with a clear configuration error.
- Missing pairing or bad signature: signed helper retries Lumi pairing once, same as task routes.
- Phone process dies or network drops: the bridge process exits; `/status` exposes `state`, `returncode`, `lastError`, `eventCount`, `lastEventAt`, `lastHeartbeatAt`, `stale`, and `stoppedBy`.
- Headers returned but no body chunks: the Node subscriber still stops at `--max-sec` because stream reads have a post-header deadline.
- Repeated starts: if a sync process is already running for the same device key, start returns the existing status instead of spawning duplicates.
- Stop: terminates the local subscriber process and closes stdout/stderr pipes.
